## Часть A. Автоматические тесты (уже зелёные — как перепроверить)

Из каталога `ui/`:


Точечно по новой фиче:

```bash
npx vitest run "src/app/(authenticated)/chat/__tests__/stream-announce.test.ts"       # сегментатор предложений
npx vitest run "src/app/(authenticated)/chat/hooks/__tests__/use-stream-announcer.test.tsx"  # хук throttle/reset/flush
npx vitest run "src/app/(authenticated)/chat/__tests__/StreamingAnnouncer.test.tsx"    # sr-only регион
```

Автотесты закрывают **DOM-контракт** (роли, aria-атрибуты, тайминги, сегментация). Чего они НЕ проверяют — реальный звук скринридера. Это Часть C.

## Часть B. Подготовка к ручному тесту

**Скринридер (обязательно для Части C):**
- **Windows:** NVDA — бесплатный, https://www.nvaccess.org/download/. Запуск: `Ctrl+Alt+N`. Выключить: `NVDA+Q` (Insert+Q).
- **macOS:** VoiceOver — встроен. Вкл/выкл: `Cmd+F5`.
- **Быстрый онлайн-вариант:** расширение Chrome «Screen Reader» (Google) — грубее, но для проверки «озвучивается / нет» годится.

**Совет:** для NVDA включите «Речевой просмотр» (NVDA menu → Инструменты → Речевой просмотр) — там текстом видно, ЧТО именно проговаривается. Это лучший способ объективно проверить озвучку.

**Браузер:** Chrome/Edge (лучшая поддержка live-region). Откройте чат, выберите агента.

## Часть C. Ручной тест скринридера (ядро)

### C1. 🎯 ГЛАВНОЕ — пофазовая озвучка ответа ассистента

**Что проверяем:** пока ассистент печатает ответ, скринридер зачитывает его **по завершённым предложениям**, а не молчит и не спамит по буквам.

**Шаги:**
1. Включите NVDA + «Речевой просмотр».
2. Отправьте запрос, провоцирующий длинный многопредложенный ответ: *«Расскажи в 4-5 предложениях, как работает индексация памяти в OPEX»*.
3. Наблюдайте за «Речевым просмотром» по мере стрима.

**Ожидается (✅):**
- По мере появления текста в просмотре речи возникают **целые предложения** — примерно раз в ~0.6 с, по мере того как каждое предложение дописывается (заканчивается на `.`/`!`/`?`/`…`).
- Незаконченное последнее предложение **не** проговаривается, пока не будет завершено.
- Когда ответ завершён, дочитывается **хвост** (последнее предложение без точки, если такое есть).
- Речь **не** повторяет весь ответ с начала на каждый токен (нет заикания/дублей).

**Красные флаги (❌):**
- Полная тишина во время стрима.
- Проговаривание по одному слову/букве.
- Повторное чтение уже озвученных предложений.

**Технически:** озвучку делает скрытый регион `role="status" aria-live="polite" aria-atomic="true"` (класс `sr-only`), а лента сообщений (`role="log"`) переведена в `aria-live="off"`, чтобы не было двойного анонса. Проверить наличие региона в DOM: DevTools → Elements → искать `Assistant response` (или `Ответ ассистента` в RU-локали) как `aria-label`.

### C2. Индикатор «думаю»

1. Отправьте запрос, дождитесь паузы перед первым текстом.
2. **Ожидается:** скринридер один раз произносит «Думаю»/«Thinking» (это `role="status"`). Двойного анонса быть не должно (лента-лог молчит).

### C3. Структура ленты (список сообщений)

1. Стрелками NVDA (или режим просмотра) пройдитесь по сообщениям.
2. **Ожидается:** лента объявляется как «журнал» (log landmark); сообщения — как **элементы списка** (`listitem`) внутри `list`. Навигация по элементам списка (`I`/`L` в NVDA) находит сообщения.

### C4. Прогресс обработки видео

1. Пришлите ссылку на видео, выберите обработчик `summarize_video`.
2. **Ожидается:** фазы (Загружаю → Транскрибирую → Составляю конспект → Сохраняю) объявляются как `role="status"`; **emoji не проговариваются** (они `aria-hidden`), звучит только локализованный текст фазы.

### C5. Разделитель сжатия контекста

1. Проведите долгую сессию до срабатывания компакции (появляется «◈ Контекст сжат · Сегмент N из M»).
2. **Ожидается:** объявляется как разделитель (`separator`) с полным текстом «Контекст сжат · Сегмент N из M» (не короче видимого).

### C6. Ошибка в потоке (error boundary)

1. Спровоцируйте сбой рендера ветки (труднее вручную; можно доверять автотесту `ThreadErrorBoundary.test.tsx`).
2. **Ожидается:** сообщение об ошибке объявляется немедленно (`role="alert"`), кнопка «Повторить» реально пере-запускает операцию (`onRetry`), а не просто пере-монтирует упавший узел.

## Часть D. Клавиатура и фокус (скринридер НЕ нужен)

Только клавиатура (`Tab`, `Shift+Tab`, `Esc`, `Enter`), мышь не трогать.

### D1. Skip-to-content
1. Обновите страницу, нажмите `Tab` один раз.
2. **Ожидается:** первым фокусируется ссылка «Перейти к содержимому» (появляется видимой в левом верхнем углу). `Enter` → фокус прыгает в `<main id="main-content">`.

### D2. ImageLightbox — ловушка фокуса
1. Откройте картинку в чате (клик по превью или `Enter` на нём).
2. **Ожидается:**
   - Фокус уходит внутрь диалога.
   - `Tab`/`Shift+Tab` **циклируют** только по кнопкам тулбара (зум−, зум+, скачать, закрыть) — фокус **не убегает** на страницу под модалкой. Проверьте особо: сразу после открытия нажмите `Tab` — фокус должен встать на первую кнопку, не выпасть наружу.
   - `Esc` закрывает; фокус **возвращается** на превью-триггер.

### D3. Поповер настроек голоса
1. Нужен активный STT-провайдер (иконка микрофона видна). Кликните шестерёнку настроек голоса.
2. **Ожидается:** фокус входит в панель (на первый слайдер); `Esc` закрывает и возвращает фокус на шестерёнку; `Tab` циклирует внутри панели.

### D4. @-упоминания — combobox
1. В поле ввода наберите `@`.
2. **Ожидается:** появляется список (`listbox`); у textarea есть `aria-controls`, указывающий на `id` списка (DevTools → у `<textarea>` атрибут `aria-controls` совпадает с `id` у `[role=listbox]`). `↑/↓` двигают активный пункт, `Enter` вставляет.

### D5. Таймстемпы по фокусу
1. На **десктопе** (ширина ≥ md): `Tab`-ните на сообщение/действия внутри бабла.
2. **Ожидается:** временная метка проявляется при `focus-within` (не только по ховеру мыши). На **мобильной** ширине метка видна всегда.
