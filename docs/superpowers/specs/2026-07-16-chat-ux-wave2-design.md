# Волна 2 UX-фич чата: поиск-палитра, букмарки, пагинация, мелкие улучшения

Дата: 2026-07-16 (ревизия 2 — после двойного адверсариального ревью спеки: технического против кода и дизайн-ревью)
Статус: одобрено (дизайн-ревью с владельцем; правки ревью внесены)
Источник: архитектурный аудит чат-UI от 2026-07-16, «волна 2». Волны 1+3 на проде (спека `2026-07-16-chat-stream-stabilization-cleanup-design.md`, финал 8f461261). Волна 4 (визуал) — отдельная будущая спека.

## Контекст

Функционально чат выше эталонов (ветвление, edit-через-fork, голос, approvals), но отстаёт в навигации по истории: поиска фактически нет (in-chat ищет по последним 100 отрендеренным, фильтр сайдбара — по загруженным 40 сессиям), сессии старше первых 40 недостижимы, вложения в композере — скрепка без миниатюры, «попробовать с другой моделью» нельзя. Серверный FTS-поиск существует и спит: `api_search_sessions` (sessions.rs:448) → `sessions::search_messages` (opex-db/sessions.rs:1597) — `plainto_tsquery('russian', $q)` по триггерной колонке `messages.tsv` (m011, GIN `idx_messages_tsv`), `ts_rank_cd`, per-agent; в результатах нет `message_id`.

## Цели

1. Глобальный поиск: палитра Ctrl+K (сессии + сообщения + переход к сообщению, включая неактивные ветки).
2. Букмарки сообщений с просмотром в палитре.
3. Infinite-scroll сайдбара — вся история сессий.
4. Превью изображений-вложений в композере.
5. Regenerate с одноразовым выбором модели (per-turn, без утечки в общий override).
6. Запоминание scroll-позиции сессии.
7. Библиотека промптов из workspace-файла.

## Не-цели

- Семантический/pgvector-поиск (v1 = существующий FTS `plainto_tsquery('russian')`).
- Пометка веток именем модели после regenerate.
- CRUD-UI и плейсхолдеры/переменные в промптах (тело вставляется как есть).
- Синхронизация scroll-позиции через БД.
- Серверный поиск в поле фильтра сайдбара (фильтр — клиентский по загруженному; поиск = палитра).

## Структура: 4 батча

| Батч | Содержание | Слои |
| --- | --- | --- |
| W2-A | Поиск-палитра Ctrl+K + переход-к-сообщению (с ветками) | Rust + UI (деплой парой) |
| W2-B | Букмарки (миграция + API + звёздочка + вкладка в палитре) | Rust + UI (деплой парой; gen-types: MessageRow ts-экспортирован) |
| W2-C | Keyset-пагинация списка сессий + infinite scroll | Rust + UI (деплой парой) |
| W2-D | Превью вложений; scroll-позиция; regenerate-с-моделью; промпты | UI + Rust (D3 — сквозная прокачка model_override) |

Порядок W2-A → W2-B → W2-C → W2-D (D независим и может идти раньше C). Работа в master, TDD, серверный гейт, деплой по батчам — процесс волны 1+3.

## W2-A. Поиск-палитра

### Бэкенд (расширение `api_search_sessions` + `sessions::search_messages`)

- Добавить `m.id AS message_id` и `s.title AS session_title` (join) в SELECT и `SearchResult`-структуру (opex-db/sessions.rs:1659; она НЕ ts-экспортирована — gen-types не затрагивается).
- Сниппет: `ts_headline('russian', m.content, plainto_tsquery('russian', $q), 'MaxWords=18, MinWords=8, ShortWord=2')` — конфиг ОБЯЗАН совпадать с писателем tsv (`'russian'`, m011), иначе подсветка не совпадёт с матчами. Маркеры `<b>…</b>` парсятся на клиенте по строковым маркерам (никакого innerHTML); стиль подсветки — токен design-system.
- Параметр `all=true`: без фильтра agent_id, каждый результат несёт `agent_id`. `?agent=` обязателен при отсутствующем/false `all` — контракт IDOR-аудита 2026-05-08 не ослабляется (single-user, общий токен и так перечисляет агентов).
- Секция «Сессии»: `title ILIKE '%q%' LIMIT 10` (title TEXT nullable, m001; таблица маленькая, индекс не нужен). Секция «Сообщения»: limit 30. Один HTTP-запрос от палитры (оба набора в одном ответе).

### Фронтенд

- `ui/src/components/chat/SearchPalette.tsx` на существующем `Dialog`; без cmdk; клавиатурная навигация по паттерну `command-autocomplete.tsx`; никаких новых сырых design-значений (ESLint `no-raw-design-values` действует; палитра — новая визуальная поверхность, волна 4 не должна её переделывать).
- Глобальный хоткей Ctrl+K / Cmd+K в `app/layout.tsx` (проверено: свободен, CodeMirror deleteLine = Shift-Mod-K, не конфликтует). `ShortcutHelp` дополняется.
- Debounce 250 мс, минимум 2 символа; тогл «по всем агентам» — состояние персистится в localStorage между открытиями; в режиме `all` КАЖДАЯ строка всех секций несёт бейдж агента (иначе кросс-агентный переход дезориентирует).
- Выбор результата: тот же агент → `selectSession` + `scrollToMessage`; другой агент → навигация `/chat?agent=…&s=…` (существующий cross-agent restore-путь), цель `{sessionId, messageId}` кладётся в store и потребляется эффектом `use-scroll-to-message` ПОСЛЕ резолва первой страницы истории (не инлайн при навигации — иначе гонка со скроллом по пустому списку).

### Переход-к-сообщению (общий механизм: поиск, букмарки, scroll-позиция)

- Хук `ui/src/app/(authenticated)/chat/hooks/use-scroll-to-message.ts` + store-цель `{sessionId, messageId}`.
- **Ветвление (ключевая политика):** целевое сообщение может лежать на неактивной ветке (edit-через-fork плодит их постоянно), а история рендерит только активный путь (`resolveActivePath`). Алгоритм: (1) если messageId есть в ЗАГРУЖЕННЫХ строках сессии, но отфильтрован как неактивная ветка — переключить `selectedBranches` на ветку с сообщением (подъём по `parent_message_id` до корня с выставлением выбора на каждой развилке), затем скролл; (2) если id в загруженном отсутствует — догружать историю `before_id`-пагинацией (`loadPreviousMessages`), после каждой страницы повторяя проверку (1); (3) потолок 20 страниц → toast «сообщение слишком глубоко» ТОЛЬКО когда id так и не появился во всём наборе строк. `renderLimit` поднимается до вхождения сообщения в отрендеренное окно.
- Скролл: `virtuosoRef.scrollToIndex` (VirtuosoHandle уже есть в MessageList) + подсветка ~2 с (токены, плавное затухание).
- Во время активного стрима переход блокируется (toast) — не воевать с autoscroll-pin.

### Критерий приёмки W2-A

Ctrl+K с любой страницы; запрос ≥2 символов ищет по текущему агенту, тогл — по всем (бейджи агентов); клик по сообщению открывает сессию, при необходимости переключает ветку и подсвечивает сообщение; сообщение на неактивной ветке НЕ даёт ложного «слишком глубоко» (vitest-тест обязателен); FTS-спецсимволы не ломают запрос (plainto_tsquery, bind-параметр).

## W2-B. Букмарки

- Миграция (следующий свободный номер): `ALTER TABLE messages ADD COLUMN bookmarked_at TIMESTAMPTZ;` + частичный индекс `CREATE INDEX idx_messages_bookmarked ON messages (bookmarked_at DESC) WHERE bookmarked_at IS NOT NULL;` (палитра открывается часто, полный скан messages недопустим).
- API: `PATCH /api/messages/{id}/bookmark?agent=` — toggle `{bookmarked: bool}`, ownership-guard инлайн-UPDATE'ом с `session_id IN (SELECT id FROM sessions WHERE agent_id=$agent)` и NotFound при `rows_affected()==0` (точный паттерн `api_message_feedback`, sessions.rs:687). `GET /api/messages/bookmarked?agent=&all=&limit=` — `JOIN sessions` (строки исчезнувших сессий исключаются автоматически), `ORDER BY bookmarked_at DESC`, отдаёт `message_id`, `session_id`, `session_title`, `agent_id`, текст-превью.
- **Превью**: только текстовые части сообщения (content мультимодальных ходов — JSON-массив частей; сырой substring дал бы JSON/base64). Нет текстовых частей → плейсхолдер («изображение»/«вложение»). Обрезка — по char-boundary ТЕКСТА (готча русских строк).
- Маршрут `GET /api/messages/bookmarked` сосуществует с `/api/messages/{id}`: axum/matchit приоритизирует статический сегмент (прецедент в этом же роутере: `/api/sessions/search` vs `/api/sessions/{id}`).
- UI: звёздочка в `MessageActions` (user и assistant), optimistic toggle; состояние приходит через `bookmarked_at` в `MessageRow` — колонку добавить в ОБА SELECT'а `get_messages_page` (offset/no-offset ветки) и в структуру. **`MessageRow` ts-экспортирован → в этом батче обязателен `make gen-types` + коммит `api.generated.ts`.**
- Палитра: пустой запрос → секция «Избранное» (до 20), клик — переход-к-сообщению W2-A (включая ветки). Если сессия результата исчезла между выборкой и кликом — toast «сессия удалена», без навигации. Тогл «по всем» действует.

### Критерий приёмки W2-B

Звёздочка ставится/снимается optimistic и переживает reload; чужой `?agent=` получает 404 (sqlx IDOR-тест); пустая палитра показывает избранное с корректными превью мультимодальных сообщений; букмарк на неактивной ветке доступен переходом.

## W2-C. Пагинация сайдбара

- Бэкенд: keyset-курсор **`(last_message_at, id) DESC`** — список сортируется по `last_message_at` (sessions.rs:69,90), колонки `created_at` у `sessions` НЕТ (только started_at/last_message_at/activity_at). Параметры `before_last_message_at` + `before_id`, tie-break по id (паттерн notifications P2). `total` остаётся.
- Фронт: `useSessions` → `useInfiniteQuery` (страница 40). **Смена формы кэша ломает трёх прямых потребителей `sessionsData.sessions`** — все три адаптируются в этом же батче: `stream-processor.ts:190` (`getQueryData<{sessions}>` — заменить на плоский селектор по pages), `page.tsx:68-70`, `MessageList.tsx:244`; плюс ~8 тест-моков. Key-only `invalidateQueries`-сайты переживают без правок.
- Virtuoso `endReached` → следующая страница; футер-спиннер; «N из M» в шапке. `endReached` продолжает пагинировать и при активном клиентском фильтре (иначе подсказка «Ctrl+K для всей истории» противоречит поведению скролла) — поведение фиксируется тестом.
- Инвалидация после create/delete/rename — `syncFirstPage`-merge (паттерн notifications P2), scroll не сбрасывается.

### Критерий приёмки W2-C

Скролл сайдбара догружает страницы до конца истории; две сессии с одинаковым `last_message_at` не дублируются и не теряются на границе страниц (sqlx-тест tie-break); rename/delete не сбрасывают позицию списка; фильтр + скролл работают одновременно.

## W2-D. Мелкие фичи

### D1. Превью вложений в композере

- Для `content[].mimeType.startsWith("image/")` чип рендерит `<ImageLightbox src={dataUrl} className="h-12 w-12 object-cover rounded" />` — компонент сам является триггером (рендерит `<img>` и открывает вьювер по клику, imperative-API нет); кнопка удаления чипа — sibling триггера, НЕ вложена в него. Не-изображения не меняются.
- Тест: чип с image-вложением рендерит миниатюру; клик открывает лайтбокс; удаление чипа работает.

### D2. Scroll-позиция сессии

- localStorage `scroll_pos:{sessionId}` — id первого видимого сообщения (Virtuoso `rangeChanged`, debounce 500 мс); пишется только при ручном отлипании от низа (`!shouldFollow`/`!isAtTail` из `useChatAutoscroll` — уже возвращаются хуком).
- Восстановление при открытии НЕ-стримящей сессии через механизм W2-A, но МОЛЧА: без подсветки, без toast'ов — если сохранённый id отсутствует в истории (сообщение/ветка удалены) или сессия стримит, тихий fallback к текущему поведению (якорь к низу). Возврат к низу очищает запись. LRU-потолок 50 записей.
- Тест: восстановление позиции; отсутствующий id → тихий fallback; стримящая сессия игнорирует запись.

### D3. Regenerate с выбором модели (самая тяжёлая часть W2-D — сквозная Rust-прокачка)

- **Нельзя** реализовывать через `engine.set_model_override`: движок — общий `Arc<AgentEngine>`, override утёк бы в параллельные/последующие ходы. Требуется per-turn поле по всему пути: `ChatSseRequest.model: Option<String>` (sse.rs:42; НЕ ts-экспортирован — gen-types не затрагивается) → bootstrap (`BootstrapOutcome`) → `pipeline/execute.rs` → `pipeline/llm_call.rs` → **`CallOptions.model_override: Option<String>`** (providers/mod.rs:173) → каждый провайдер (openai, anthropic, google, http, claude_cli) подставляет его вместо `current_model()` в тело запроса. Не персистится, `model_overrides` не трогается.
- Обязательный тест: после хода с `model` `engine.current_model()` НЕ изменился; параллельный ход без `model` использует штатную модель.
- Фронт: `model` прокидывается через **оба** пути — `regenerate()` И `regenerateFrom(messageId)` → `forkAndStream` → `renderer.sendTurn`; `sendTurn` вызывается с 3 сайтов — параметры переводятся на options-объект вместо 6-го позиционного. Сплит-кнопка у ReloadButton (клик = как сейчас; шеврон = дропдаун моделей). Источник списка — модели профиля ТЕКУЩЕГО агента (как в композере); в мультиагентной сессии regenerate и так перегенерирует ход текущего агента — семантика не меняется.

### D4. Библиотека промптов

- Файл `workspace/prompts.md`: секции `## Заголовок` + тело. Чтение: `apiGet<WorkspaceFile>('/api/workspace/prompts.md')` → `.content` (именно так читает Workspace-вьювер; signed-URL путь — только для бинарных). Парсер `lib/prompts.ts`, React Query кэш.
- **Memory-индексация**: root-level `.md` в workspace индексируется watcher'ом в `memory_chunks` — `"prompts.md"` добавляется в `MEMORY_INDEX_EXCLUDE_FILES` (workspace.rs:27), чтобы тексты промптов не загрязняли память агентов.
- Композер: слэш-автокомплит расширяется секцией «Промпты» ПОСЛЕ серверных команд. Это не смена data-source, а расширение контракта `command-autocomplete.tsx`: item-модель получает вид записи (command | prompt; промпты рендерятся без `/`-префикса — исключает шэдоуинг реальных команд), `onPick` различает тип — команда работает как сейчас, промпт → родитель ЗАМЕНЯЕТ текст поля телом промпта (промпт = стартовый шаблон), курсор в конец, БЕЗ отправки.
- Welcome-экран: первые 3 промпта вместо статичных подсказок (fallback на статику при отсутствии/пустоте файла). Файл создаётся лениво; в scaffold агентов сид не добавляется (файл общий на workspace). (Отмечено ревью как кандидат на вырезание при сжатии скоупа — оставляем: дёшево при готовом `lib/prompts.ts`.)
- Тест: парсер (заголовки/тела/пустой файл); выбор промпта заменяет текст и не отправляет; команда рядом с одноимённым промптом не шэдоуится.

### Критерий приёмки W2-D

Миниатюры вложений с лайтбоксом; scroll-позиция восстанавливается и не мешает стриму; regenerate с моделью X создаёт ветку от модели X, следующий ход — штатной моделью, `current_model()` не изменился; промпты доступны из `/`-меню и welcome, файл не индексируется в память.

## Тестирование и деплой

- Процесс волны 1+3: TDD, vitest из `ui/`, Rust — серверный контур (BIN-таргет, `bash -lc`, PG opex_test:5434), clippy `-D warnings`.
- `make gen-types` без диффа перед КАЖДЫМ push; W2-B меняет ts-экспортированный `MessageRow` — regen обязателен в самом батче.
- sqlx: search (`message_id`+`all`+headline c 'russian'); bookmark toggle + IDOR; keyset tie-break; D3 non-persistence.
- vitest: палитра (debounce/секции/клавиатура/бейджи агентов); переход-к-сообщению ВКЛЮЧАЯ неактивную ветку и потолок-20; infinite sessions merge + фильтр-со-скроллом; парсер промптов + вставка; сплит-кнопка (оба пути regenerate); D1/D2 по критериям выше.
- Деплой: все четыре батча — парой (каждый трогает Rust). E2E-смоук после каждого; ручной E2E палитры на ветвистой длинной сессии.

## Риски

| Риск | Митигация |
| --- | --- |
| Переход к сообщению на неактивной ветке ломается/врёт «слишком глубоко» | Политика веток в W2-A: сначала поиск по всем строкам + переключение selectedBranches; toast только при реальном отсутствии; обязательный тест |
| ts_headline дорог на огромных телах сообщений | MaxWords-кап; GIN-индекс обслуживает WHERE; limit 30 на секцию |
| FTS-спецсимволы в запросе | plainto_tsquery (plain-text парсер) + bind-параметр — как в существующем search_messages |
| Одноразовый model персистится или утекает в параллельный ход | CallOptions-прокачка вместо set_model_override; тесты non-persistence + параллельного хода |
| Смена формы кэша sessions ломает потребителей | Три прямых потребителя перечислены и адаптируются в W2-C; syncFirstPage-merge; тесты |
| Восстановление scroll воюет с autoscroll | Только НЕ-стримящая сессия; тихие fallback'и; autoscroll-pin в приоритете |
| prompts.md загрязняет память агентов | MEMORY_INDEX_EXCLUDE_FILES += prompts.md |
