# Спека: Server-authoritative чат-стрим — тонкий клиент, snapshot-протокол на SSE

**Дата:** 2026-07-15
**Статус:** дизайн согласован (брейншторм + независимое индустриальное ревью)
**Триггер:** при обновлении страницы посреди/после хода «тот же запрос шляется» — клиентский resume переигрывает стрим по собственным правилам; запрос владельца — «UI должен стать транслятором».

## 1. Проблема

Класс багов на границах live↔history порождён тем, что **клиент владеет жизненным циклом стрима**:

- refresh → клиент сам решает звать `GET /api/chat/{id}/stream`, тащит `Last-Event-ID`, переигрывает буфер «как живой»;
- live-сообщения и REST-история накладываются → контентная эвристика `chat-overlay-dedup.ts`;
- `activeSessionIds`, connection-фазы, reconnect-машина — ~1600 LoC стора + ~900 LoC stream-обвязки + ~800 LoC actions, десятки состояний и гонок.

Каждый сценарий (refresh, фоновая вкладка, обрыв сети, вторая вкладка) — отдельная ветка. Отдельные ветки = отдельные баги.

## 2. Решение одним абзацем

**Сервер владеет жизненным циклом; клиент — идемпотентный рендерер.** Любое подключение к сессии проходит **один и тот же код-путь**: клиент открывает стрим → сервер присылает авторитетный snapshot-конверт текущего хода (полный типизированный event-log из `StreamRegistry`) + явную границу history↔live → дальше live-события. Клиент применяет конверт батчем (без анимации), live — по порядку, всё идемпотентно по id. Нет спецслучаев → нет класса багов. Транспорт — **существующий SSE** (не WebSocket — см. §3).

## 3. История выбора и обоснование транспорта

В брейншторме первоначально был выбран WebSocket-на-сессию. **Независимое ревью в чистой сессии** (индустриальный анализ: ChatGPT, Claude.ai, Vercel AI SDK `experimental_resume`, LibreChat Resumable Streams, LobeChat, HF chat-ui, Open WebUI, Phoenix LiveView, Convex, Linear) показало:

- токен-стрим у всех LLM-чатов — **SSE**; WS — только у Open WebUI;
- resumable-паттерн сошёлся к одному рецепту: сервер буферизует дельты, на реконнекте отдаёт **агрегированный sync-снапшот + границу**, дальше live (LibreChat — точный прецедент);
- корень наших багов — не транспорт, а **владение resume/replay**; фикс транспорт-независим;
- `/v1/chat/completions` обязан остаться SSE → выбор WS навсегда обрекает на **два конвертера событий**, которые будут расходиться (новый источник багов — противоречит цели «раз и навсегда»);
- mobile-background не решается автоматически ни SSE, ни WS (vercel/ai#11865) — в обоих случаях нужен клиентский visibility-хук.

**Решение: SSE + server-authoritative snapshot.** WS-на-сессию зафиксирован как **эскалационный путь** (§12): snapshot-слой транспорт-независим, при доказанной недостаточности SSE на мобильном меняется только «труба».

## 4. Протокол

### 4.1 Отправка хода

`POST /api/chat` больше **не стримит ответ**. Он валидирует, персистит user-сообщение, спавнит движок (как сейчас) и сразу отвечает `202 {"session_id": …}`. Весь стриминг — только через GET-стрим (§4.2). Это делает путь «отправил ход» и путь «переподключился» **одним и тем же** кодом на клиенте: после POST клиент просто (пере)открывает стрим.

- Задержка 202: boundary (= user_message_id) появляется после bootstrap'а, который включает `enrich_message_text` (голос-транскрипция/vision/URL-fetch) и компакцию — на ходах с вложениями/URL POST может занять секунды. Принято: optimistic echo клиента закрывает паузу визуально; на чисто текстовых ходах — практически один RTT. События до подключения GET не теряются — `StreamRegistry` буферизует с seq 0, snapshot их доставит.
- `POST /api/chat/{id}/abort` — без изменений (явный Stop).

### 4.2 Единый стрим: `GET /api/chat/{session_id}/stream`

Заменяет собой сегодняшнюю пару «стрим из POST + resume.rs». На каждое подключение сервер шлёт:

```text
sync_begin { boundary_message_id, run_status, truncated }
  … полный replay буфера активного хода (те же типизированные события,
    что сегодня: start/text-delta/tool-input-*/tool-output-available/
    approval/file/rich-card/step-start/…; каждое с SSE id = seq) …
sync_end { last_seq }
  … live-события по мере генерации (id = seq) …
finish { … }            ← как сегодня
```

- **`boundary_message_id`** = id user-сообщения активного хода (персистится в bootstrap до запуска). Контракт рендера: REST-история рисуется **вплоть до boundary включительно**; всё после boundary — live-состояние из snapshot+live. Контентный дедуп не нужен.
- **Нет активного хода** (run завершён/нет) → `sync_begin{run_status:"finished", boundary_message_id:null}` + `sync_end` и стрим закрывается. Клиент рисует чисто REST-историю. **Это и есть фикс исходного бага**: refresh после завершённого хода не переигрывает ничего.
- **`Last-Event-ID` упраздняется.** Реплей всегда полный (с seq 0): клиент на каждом подключении **сбрасывает live-состояние хода и строит заново из конверта** — идемпотентность через «rebuild from authoritative log», а не через клиентский учёт позиций.
- **`truncated: true`** — деградация при переполнении буфера (§5.3): клиент берёт частичный текст ассистента из REST (`streaming_db` уже персистит его инкрементально), затем применяет хвост буфера и live.
- Конвертер `StreamEvent → SSE JSON` — **единственный, существующий** (`sse_converter.rs`); типы wire-событий не меняются, добавляются только `sync_begin`/`sync_end`.

### 4.3 Что удаляется из протокола

`GET …/stream` как «resume-опция» (теперь это единственный путь), `Last-Event-ID`, стриминг из POST-ответа, `204`-семантика resume (заменена пустым sync-конвертом).

## 5. Сервер (opex-core)

### 5.1 Единый stream-handler

`sse.rs` + `resume.rs` сливаются в один handler GET-стрима: атомарный `subscribe` к `StreamRegistry` (снапшот+broadcast уже атомарны — stream_registry.rs), обёртка конверта `sync_begin/sync_end`, затем relay broadcast. `POST /api/chat` сокращается до «bootstrap + spawn + 202». Спавн-путь движка (`handle_sse` → pipeline) не меняется.

### 5.2 `StreamRegistry` — расширения

- хранит `boundary_message_id` (id user-сообщения run'а — передаётся при регистрации стрима из bootstrap);
- `run_status` для «нет активного стрима» отвечается из DB (как сейчас в resume.rs), но решение принимает **сервер**, клиенту приходит готовый пустой конверт;
- сохранение стрима в реестре до `finish` + грейс (как сейчас), `is_finished` — уже есть.

### 5.3 Ёмкость буфера

Буфер должен вмещать полный ход (single-user система). `BROADCAST_CAPACITY`/ring поднять до конфигурируемого значения (default 10 000 событий — с запасом покрывает длинный tool-ход); при переполнении — существующее поведение «buffer full: broadcast only» дополняется флагом `truncated` в `sync_begin`. Fallback-контент — из инкрементально персистимого partial (`upsert_streaming_append`), который уже есть.

### 5.4 Инварианты сервера

- **Разрыв GET-стрима ≠ abort движка** (сохранённый инвариант от 2026-07-13: закрытая вкладка не убивает работу). Отписка = detach подписчика.
- Abort — только явный `POST /abort`.
- `/v1/chat/completions`, `/v1/models`, openai-compat — **не трогаются**.
- Канальный путь (`channel_ws`) — не трогается.

## 6. Клиент (ui)

### 6.1 Контракт владения (фиксируется как норма кода)

| Владеет СЕРВЕР | Владеет КЛИЕНТ |
| --- | --- |
| жизненный цикл хода, run_status | рендер: конверсия событий → parts (существующие ~31 кейс `stream-processor`) |
| порядок и seq событий, содержимое snapshot | ввод, композер, T4-очередь голосовых во время хода |
| решение «что реплеить» на каждом подключении | optimistic-echo user-сообщения после POST |
| граница history↔live (`boundary_message_id`) | visibility/staleness-триггер переподключения |
| финализация/persist | scroll/focus/анимации |

### 6.2 Что удаляется (~1300+ LoC и, главное, состояния)

- `stream-reconnect.ts`, reconnect/`Last-Event-ID`/connection-management в `streaming-renderer.ts`;
- `chat-overlay-dedup.ts` целиком; live-merge-логика `chat-history.ts`;
- `activeSessionIds`, resume-решения, `run_status`-гейтинг в navigation/ChatThread;
- фазовая машина сжимается до `idle | submitted | streaming | error` (`error` сохраняется — на нём стоят anti-clobber guard коммита и error-discard очереди; `reconnecting` фолдится в `submitted`+staleness-переоткрытие, `complete` — в `idle`).

### 6.3 Что появляется (маленькое)

- `chat-stream.ts`: открыть `EventSource`, применить конверт батчем (одна commit-отрисовка, без анимации по событию), live — как сейчас; на `error`/staleness → закрыть и переоткрыть (тот же путь);
- visibility-хук: `visibilitychange`→foreground и staleness-таймер (ход активен, а событий нет дольше 15 сек — default, конфигурируемо) → переоткрыть стрим. **Обязателен** — сервер-авторитетность не отменяет этот клиентский триггер (урок vercel/ai#11865);
- id-based handoff: на `finish` — invalidate REST-истории; когда refetch вернул сообщения хода (по message id), live-состояние хода очищается. До того — рендер live-копии (визуально идентично).

### 6.4 Сохраняемое поведение

- Модель `ChatMessage`/`MessagePart` не меняется → `MessageList`, `tts-speaker` (голосовой UX), approvals/clarify-карточки работают без правок; **1356 vitest, включая голосовые, зелёные — критерий приёмки**.
- Optimistic echo: после POST клиент сразу рисует user-сообщение; на refresh оно приходит из REST-истории (boundary включает его в history-сторону).
- Multi-tab: N подписчиков одного broadcast; approvals, отвеченные в другой вкладке, приедут в snapshot при следующем подключении/в live.

## 7. Cutover

Транспорт остаётся SSE, поэтому cutover мягче, чем при WS:

1. Сервер: новый конверт + `POST=202` деплоятся **одновременно** с новым UI (single-user система; деплой = `server-deploy.sh` + `deploy-ui.sh` в одном окне).
2. Клиентская машина (resume/dedup/overlay) удаляется в том же релизе — по выбору владельца «чистый cutover».
3. Страховка отката: `deploy-ui.sh` хранит 3 релиза (мгновенный симлинк-флип), но старый UI требует старого POST-стриминга → откат = откат обоих (git revert + redeploy). Принято осознанно.

## 8. Тесты

- **Rust:** конверт (sync_begin/end, boundary, пустой конверт при finished, truncated); атомарность subscribe+snapshot (есть); «GET-разрыв не отменяет движок»; POST=202+спавн; abort-путь.
- **UI (vitest):** батч-применение конверта идемпотентно (двойное подключение = идентичное состояние); rebuild-from-log при переподключении посреди tool-хода (tool-input/approval/file в конверте); boundary-фильтр REST-истории; handoff на finish; visibility/staleness-переоткрытие; T4-очередь и голосовой speaker — существующие тесты без изменений.
- **E2E-чеклист (прод):** refresh ПОСЛЕ завершённого хода → мгновенно чистая история, ничего «не шляется» (исходный баг); refresh ПОСРЕДИ tool-хода → снапшот восстанавливает текст+tool-карточки без дублей, live продолжается; закрыть вкладку на 2 мин → вернуться → ход дорешался, ответ на месте; телефон: свернуть на минуту посреди генерации → развернуть → догнал; вторая вкладка той же сессии; Stop работает; голосовой ход со стриминговой озвучкой не сломан.

## 9. Риски (из независимого ревью, приняты в дизайн)

1. **Шов — это tool/approval/file-события, не текст.** Снапшот = полный event-log (не отдельная серверная редукция) — редукция остаётся одна, клиентская, та же для live; иначе два аггрегатора разойдутся так же, как разошлись бы два транспорта.
2. **Mobile-background не решается «авторитетным сервером»** — visibility-хук обязателен и входит в клиентский контракт (§6.1/§6.3).
3. **«Тонкий клиент» ≠ нулевой клиент.** Предел — ~200 строк состояния + неустранимая event→parts конверсия. Попытка «ещё тоньше» = новый класс багов.

## 10. Вне объёма

- WS-транспорт — эскалационный путь: та же snapshot-логика, замена «трубы» (`channel_ws`-паттерн + ticket-auth готовы), триггер — доказанная на практике недостаточность SSE+visibility на мобильном.
- Мультиплекс-сокет на вкладку, server-driven UI (несовместимо со статическим Next-экспортом) — отклонены.
- Серверная очередь веб-ходов (аналог channel `session_queue`) — не в v1: клиентская T4-очередь только что отлажена.
- Каналы (Telegram/Discord), openai-compat — не затрагиваются.
