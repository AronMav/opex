# Волна 2 UX-фич чата: поиск-палитра, букмарки, пагинация, мелкие улучшения

Дата: 2026-07-16
Статус: одобрено (дизайн-ревью с владельцем)
Источник: архитектурный аудит чат-UI от 2026-07-16, «волна 2» (UX-пробелы против эталонов ChatGPT/Claude.ai). Волны 1+3 (баги+уборка) реализованы и на проде (спека `2026-07-16-chat-stream-stabilization-cleanup-design.md`, финал 8f461261). Волна 4 (визуал) — отдельная будущая спека.

## Контекст

Функционально чат уже выше эталонов (ветвление, edit-через-fork, голос, approvals), но отстаёт в навигации по истории: поиска фактически нет (in-chat ищет по последним 100 отрендеренным, фильтр сайдбара — по загруженным 40 сессиям), сессии старше первых 40 недостижимы через UI, вложения в композере — скрепка без миниатюры, «попробовать с другой моделью» нельзя. Серверный FTS-поиск по сообщениям уже существует (`api_search_sessions` → `sessions::search_messages`, per-agent, с ранжированием), но спит: UI его не вызывает, а в результатах нет `message_id`.

## Цели

1. Глобальный поиск по истории: палитра Ctrl+K (сессии + сообщения, переход к найденному сообщению).
2. Букмарки сообщений с просмотром в палитре.
3. Infinite-scroll сайдбара — доступ ко всей истории сессий.
4. Превью изображений-вложений в композере.
5. Regenerate с одноразовым выбором модели.
6. Запоминание scroll-позиции сессии.
7. Библиотека промптов из workspace-файла.

## Не-цели

- Семантический/pgvector-поиск по сообщениям (v1 = существующий FTS; memory-поиск — отдельная система).
- Пометка веток именем модели после regenerate (сравнение — через BranchNavigator).
- CRUD-UI для промптов (редактирование — Workspace-вьювер).
- Синхронизация scroll-позиции через БД (localStorage достаточно).
- Серверный поиск в поле фильтра сайдбара (поиск = палитра; фильтр остаётся клиентским по загруженному).

## Структура: 4 батча

| Батч | Содержание | Слои |
| --- | --- | --- |
| W2-A | Поиск-палитра Ctrl+K + механизм перехода-к-сообщению | Rust + UI (деплой парой) |
| W2-B | Букмарки (миграция + API + звёздочка + вкладка в палитре) | Rust + UI (деплой парой) |
| W2-C | Курсорная пагинация списка сессий + infinite scroll | Rust + UI (деплой парой) |
| W2-D | Превью вложений; scroll-позиция; regenerate-с-моделью; промпты | UI + одно малое Rust-поле |

Порядок W2-A → W2-B (палитра — носитель букмарок) → W2-C → W2-D (фичи D независимы и могут идти раньше C). Работа в master, TDD, серверный гейт, деплой по батчам — процесс волны 1+3.

## W2-A. Поиск-палитра

### Бэкенд (расширение `api_search_sessions`, `crates/opex-core/src/gateway/handlers/sessions.rs:448` + `sessions::search_messages` в opex-db)

- В результаты добавить `message_id` (uuid строки messages) и `session_title` (join на sessions.title) — без них невозможен переход к сообщению и осмысленный заголовок результата.
- Параметр `all=true`: поиск без фильтра по agent_id, каждый результат несёт `agent_id`. Безопасность: инстанс single-user, общий bearer-токен и так видит всех агентов перебором `?agent=`; `?agent=` остаётся обязательным при `all` отсутствующем/false (контракт IDOR-аудита 2026-05-08 не ослабляется — `all` просто явный новый режим).
- Сниппет: `ts_headline` (FTS-подсветка `<b>…</b>` → на клиенте рендерится highlight-спаном, HTML не вставляется как innerHTML — парсится по маркерам) вместо сырого `content`.
- Отдельная лёгкая выборка сессий по заголовку: `title ILIKE '%q%'` (лимит 10) — палитра показывает секцию «Сессии» над «Сообщениями». Реализуется в том же handler'е (один HTTP-запрос от палитры).

### Фронтенд

- Новый `ui/src/components/chat/SearchPalette.tsx` на существующем `Dialog` (`components/ui/dialog.tsx`). Без новой зависимости (cmdk НЕ добавляем): клавиатурная навигация (стрелки/Enter/Esc) по паттерну `command-autocomplete.tsx`.
- Глобальный хоткей Ctrl+K / Cmd+K, регистрируется в `app/layout.tsx` (палитра доступна с любой страницы; открытие с не-чат-страницы при выборе результата ведёт на `/chat?agent=…&s=…`). Ctrl+K свободен (проверено — в коде не привязан); `ShortcutHelp` дополняется.
- Debounce 250 мс, минимум 2 символа; тогл «по всем агентам» (переключение — повторный запрос с `all=true`); индикатор загрузки; пустой результат — явное состояние.
- Секции: «Сессии» (по title) и «Сообщения» (FTS, сниппет + агент + дата). Клик/Enter: `selectSession` (при необходимости — со сменой агента, существующий cross-agent путь `?s=` из restore-машины) + `scrollToMessage(messageId)` для результатов-сообщений.

### Переход-к-сообщению (общий механизм для поиска, букмарок и scroll-позиции)

- Новый хук `ui/src/app/(authenticated)/chat/hooks/use-scroll-to-message.ts` + store-действие: цель = `{sessionId, messageId}`.
- Алгоритм: если messageId есть в загруженных страницах — Virtuoso `scrollToIndex` по индексу в отрендеренном списке (учесть `renderLimit`: при необходимости поднять лимит до вхождения сообщения); если нет — догружать историю существующей `before_id`-пагинацией (`loadPreviousMessages`) до нахождения id (максимум N страниц — потолок 20, затем честная ошибка-toast «сообщение слишком глубоко»), после чего скролл.
- Подсветка найденного сообщения ~2 с (класс с плавным затуханием, токены design-system).
- Во время активного стрима переход блокируется (toast), чтобы не воевать с autoscroll-pin.

## W2-B. Букмарки

- Миграция (следующий свободный номер на момент реализации): `ALTER TABLE messages ADD COLUMN bookmarked_at TIMESTAMPTZ` (NULL = не букмарк). Отдельной таблицы не нужно (YAGNI: один пользователь, нет папок/тегов).
- API: `PATCH /api/messages/{id}/bookmark?agent=` — toggle (тело `{bookmarked: bool}`), с `verify_session_agent`-проверкой владения через сессию сообщения (образец — `api_message_feedback`); `GET /api/messages/bookmarked?agent=&all=&limit=` — последние по `bookmarked_at DESC` с `session_id`, `session_title`, `content`-превью (обрезка на char-boundary!), `agent_id`.
- UI: звёздочка в `MessageActions` (для user и assistant сообщений; состояние из истории — messages row несёт bookmarked_at через существующий messages-эндпоинт), optimistic toggle.
- Палитра: при пустом запросе показывает секцию «Избранное» (до 20), клик — переход-к-сообщению из W2-A. Тогл «по всем агентам» действует и здесь.

## W2-C. Пагинация сайдбара

- Бэкенд: `api_list_sessions` получает курсор `before_created_at` + `before_id` (keyset по `(created_at, id) DESC` с tie-break по id — паттерн notifications P2, m085-эпоха). `total` уже возвращается — остаётся.
- Фронт: `useSessions` → `useInfiniteQuery` (страница 40); Virtuoso списка сессий `endReached` → следующая страница; футер-спиннер; «N из M» в шапке списка. Инвалидация после create/delete/rename — сброс до первой страницы с сохранением scroll (паттерн `syncFirstPage`-merge из notifications).
- Клиентский фильтр (`sessionFilter`) фильтрует ЗАГРУЖЕННОЕ + подсказка «искать по всей истории — Ctrl+K» под полем при активном фильтре.
- Multi-select delete работает по загруженным (как сейчас).

## W2-D. Мелкие фичи

### D1. Превью вложений в композере

- В чипах вложений: для `content[].mimeType.startsWith("image/")` — миниатюра 48×48 (`object-cover rounded`) из уже имеющегося data-URL (`att.content[].data`), клик — существующий `ImageLightbox`; удаление чипа — как сейчас. Не-изображения не меняются.

### D2. Scroll-позиция сессии

- localStorage `scroll_pos:{sessionId}` — id первого видимого сообщения (Virtuoso `rangeChanged`, debounce 500 мс); пишется только при ручном отлипании от низа (существующий детектор autoscroll).
- При открытии сессии: если запись есть и сессия НЕ стримит — восстановление через механизм перехода-к-сообщению (без подсветки); иначе — текущее поведение (якорь к низу). Возврат к низу вручную/новым сообщением очищает запись.
- Ограничение объёма: хранить максимум 50 записей (LRU-очистка).

### D3. Regenerate с выбором модели

- Бэкенд: опциональное поле `model: Option<String>` в теле `POST /api/chat` — одноразовый override ровно на этот ход: прокидывается через bootstrap в вызов LLM ПОВЕРХ agent/profile/persistent-override резолва, НЕ персистится и не трогает `model_overrides`. Валидация: непустая строка; неизвестная модель отдаст ошибку провайдера штатно.
- Фронт: `ReloadButton` в `MessageActions` становится сплит-кнопкой: клик — regenerate как сейчас; шеврон — дропдаун моделей (источник — тот же список, что у существующего `ModelDropdown` композера) → `regenerate({model})`. `stream-control.ts` `forkAndStream`/`sendTurn` прокидывают `model` в тело. После хода следующие ходы идут обычной моделью.

### D4. Библиотека промптов

- Файл `workspace/prompts.md`: секции `## Заголовок` + тело до следующего `##`. Читается существующим workspace-API (signed URL / прямое чтение как в Workspace-вьювере), парсится на клиенте (`lib/prompts.ts`), кэш React Query с инвалидацией по фокусу.
- Композер: в существующем слэш-автокомплите добавляется секция «Промпты» (после серверных команд); выбор промпта ВСТАВЛЯЕТ его тело в поле ввода (не отправляет), курсор в конец.
- Welcome-экран: вместо 3 статичных подсказок — первые 3 промпта из файла (fallback на текущую статику, если файла нет или он пуст).
- Файл создаётся лениво: если отсутствует, welcome/автокомплит просто не показывают секцию. В scaffold агентов сид-пример НЕ добавляется — файл общий на workspace, не пер-агентный.

## Тестирование и деплой

- Процесс волны 1+3: TDD, vitest строго из `ui/`, Rust — серверный контур (BIN-таргет, `bash -lc`, PG opex_test:5434), clippy `-D warnings`, `make gen-types` без диффа перед push (в W2-D меняется тело ChatRequest — если оно в ts-rs экспорте, регенерировать!).
- sqlx-тесты: search с `message_id`+`all`+headline; bookmark toggle + IDOR-guard (чужой agent → 4xx); keyset-курсор с tie-break (две сессии с одинаковым created_at).
- vitest: палитра (debounce, секции, клавиатура), scroll-to-message (догрузка до потолка + ошибка), infinite sessions merge, парсер prompts.md, сплит-кнопка regenerate.
- Деплой: W2-A/B/C — парой; W2-D — парой (D3 трогает Rust). E2E-смоук после каждого батча; ручной E2E палитры и перехода-к-сообщению на длинной сессии.

## Риски

| Риск | Митигация |
| --- | --- |
| Переход к очень глубокому сообщению = лавина запросов истории | Потолок 20 страниц + toast; страницы догружаются последовательно |
| ts_headline с пользовательским запросом (спецсимволы FTS) | plainto_tsquery/websearch_to_tsquery (как в существующем search_messages — сверить и переиспользовать), запрос — bind-параметр |
| Одноразовый model в POST /api/chat случайно персистится | Тест: после хода с model engine.current_model() не изменился; поле не пишется в model_overrides |
| Инвалидация infinite-списка сессий ломает scroll | syncFirstPage-merge паттерн из notifications P2 + тест merge |
| Восстановление scroll-позиции воюет с autoscroll | Восстановление только при НЕ-стримящей сессии; autoscroll-pin имеет приоритет |
