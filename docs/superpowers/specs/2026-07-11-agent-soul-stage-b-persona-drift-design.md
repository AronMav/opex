# Agent Soul — Stage B: Persona Drift (within-session monitor + auto A-anchor)

**Дата:** 2026-07-11
**Статус:** design approved (устно), ждёт ревью спеки
**Программа:** этап B из трёх («душа агента»). Фундамент (этап A+D) задеплоен 2026-07-10.
**Исследовательская база:** [docs/research/2026-07-09-agent-soul-research.md](../../research/2026-07-09-agent-soul-research.md), находки 6–9.
**Опирается на:** foundation-спеку [2026-07-09-agent-soul-foundation-design.md](2026-07-09-agent-soul-foundation-design.md) (раздел «Зависимости следующих этапов», этап B).

## Цель

Персона-дрифт — быстрый и повсеместный (LLaMA2 теряет персону за ~8 ходов; 17/23 фронтир-моделей дрифтуют на длинных сессиях; статичный SOUL.md/IDENTITY.md его НЕ держит — находки 6, 7). Этот этап добавляет **активный контур стабилизации личности внутри сессии**: дешёвый embedding-only детектор дрифта + автоматическая коррекция инъекцией A-anchor.

### Решения, зафиксированные с оператором

| Вопрос | Решение |
| --- | --- |
| Тип дрифта | Внутрисессионный (классика: агент перенимает персону собеседника, теряет регистр за длинный диалог). Долгосрочный SELF.md-дрифт — вне v1. |
| Метод детекции | Пассивный anchor-косинус (Nautilus Compass: эмбеддинг поведения агента против курируемых якорей, AUC 0.83) — БЕЗ LLM-вызовов, целиком в хуках. |
| Коррекция | Авто A-anchor (~80-токенное напоминание идентичности), инъекция в context_builder при превышении порога. High-confidence в исследовании (находка 9). |

### Вне скоупа

- **Долгосрочный дрифт** (эволюция SELF.md уводит личность от ядра за недели) — отдельный контур на heartbeat-пробах, кандидат на B2.
- **ECP** (Egocentric Context Projection) — валидирован только на ≤25-ходовых симуляциях; кандидат на B2, если A-anchor окажется недостаточен на живом трафике.
- **Активные identity-пробы** (SPASM: агент отвечает на вопросы) — точнее, но стоит LLM-вызовов; не в v1.
- Этап C (инициатива/планирование) — отдельная спека.

## 1. Архитектура

Пер-ходовой embedding-only контур в `context_builder`, полностью opt-in через `[agent.drift]`, переиспользующий toolgate-embeddings и identity-доки агента (SOUL/IDENTITY/SELF). **Никаких новых таблиц. Никаких LLM-вызовов в горячем пути** (LLM зовётся один раз при генерации якорей). Fail-soft везде.

Поток данных:

1. **Загрузка агента / первое использование:** прочитать `DRIFT_ANCHORS.md`; если его нет и `enabled` — сгенерировать якоря + A-anchor из identity-доков (1 LLM-вызов, разово), записать файл. Заэмбедить якоря через toolgate, закэшировать эмбеддинги в памяти (пер-агент).
2. **Каждый ход (при `history_len ≥ min_history`):** заэмбедить последний ответ(ы) агента → `drift_score` = функция косинусов к позитивному/негативному центроидам якорей. +1 embedding.
3. **Коррекция:** если `drift_score > threshold` — вставить закэшированный A-anchor синтетическим user-ходом между префиксом сессии и текущим ходом. Залогировать; отразить в `ContextBreakdown`.

## 2. Модель данных и хранение

### `DRIFT_ANCHORS.md` (workspace-файл, human-editable)

`workspace/agents/{Name}/DRIFT_ANCHORS.md` — по образцу SELF.md (файловая identity-модель фундамента). Секции:

```markdown
# DRIFT ANCHORS — {AGENT_NAME}

> Якоря для детекции персона-дрифта. Позитивные = агент «в характере»;
> негативные = дрифт (эхо собеседника, сикофантия, потеря регистра).
> Дизайн якорей — главный фактор точности; правьте под реальный трафик.

## В характере (positive)
- <task-shaped предложение, как агент формулирует в своём стиле>
- ...

## Дрифт (negative)
- <предложение в чужом/дрифтовом стиле>
- ...

## A-anchor
<~80-токенное напоминание идентичности + один пример ответа в характере>
```

- **Генерация (разово, при первом включении):** LLM получает SOUL.md + IDENTITY.md + SELF.md и промпт «сгенерируй N позитивных task-shaped якорей (стиль/ценности/регистр этого агента), N негативных (дрифт), и один A-anchor». Записывается в файл. Оператор может редактировать/регенерировать (`POST /api/agents/{name}/drift/regenerate`).
- **Негативные якоря:** агент-специфичные (из генерации) + общий базовый набор (сикофантия, «как скажешь», эхо-персона) — константа в коде, конкатенируется.
- **Кол-во:** ~10 позитивных + ~10 негативных (константа, не конфиг — YAGNI). Дизайн-фактор — качество, не количество.

### Кэш эмбеддингов (в памяти, пер-агент)

`Arc<RwLock<Option<CachedAnchors>>>` — поле на `AgentConfig` (рядом с `soul_runtime` этапа A). `CachedAnchors { positive: Vec<Vec<f32>>, negative: Vec<Vec<f32>>, a_anchor: String }`. Заполняется **лениво** при первой drift-проверке (эмбеддит ~20 якорей одним `embed_batch`, кэширует). Если toolgate недоступен — кэш пуст, детекция пропускается, ретрай на следующем ходу. Файл изменился (mtime) → инвалидация кэша + пере-эмбеддинг.

### Защита файла

`DRIFT_ANCHORS.md` добавляется в:
- оба skip-листа `load_workspace_prompt` / `load_workspace_prompt_excluding_claude_md` (как SELF.md) — **не грузится в промпт** дословно;
- **НЕ защищён** (v1-решение): не в `IDENTITY_FILES`, не в `is_read_only`. Оператор и агент могут читать/писать/удалять его тулами. Обоснование: содержимое **никогда не попадает в промпт** (только эмбеддится), поэтому это не канал prompt-injection; оператор-редактируемость якорей важнее (дизайн якорей — главный фактор точности). Удаление файла → регенерация при следующем использовании.

## 3. Детектор дрифта (алгоритм)

Чистая функция (тестируемо без БД, без сети):

```text
positive_centroid = mean(normalize(positive_embeddings))
negative_centroid = mean(normalize(negative_embeddings))
recent = normalize(embed(last_assistant_turns))   # последние 1-2 ответа агента, склеенные
drift_score = cos(recent, negative_centroid) − cos(recent, positive_centroid)
```

- `drift_score ∈ [−2, 2]`; выше = дальше от характера, ближе к дрифту. Порог `threshold` (дефолт ~0.05, подбирается на живом трафике).
- Что эмбеддится (v1): **последний ответ агента** из истории — прямой сигнал, куда дрейфует поведение. Если ответов ещё нет (первый ход) — детекция пропускается. (Склейка нескольких ответов / weighted top-k — кандидаты на B2.)
- **v1 использует центроид** (простой, тестируемый). Weighted top-k mean (как Compass) — кандидат на B2, если центроид окажется шумным на живом трафике.
- Пропуск, если `history_len < min_history` (дефолт 6) — на коротких сессиях дрифта нет, экономим embedding.

## 4. Корректор (A-anchor)

При `drift_score > threshold`:

- A-anchor (из `CachedAnchors.a_anchor`, ~80 токенов) вставляется в `messages` **синтетическим user-ходом непосредственно перед текущим user-сообщением** (последним в истории) — «между префиксом сессии и текущим ходом» (находка 9). Роль user (не system), т.к. механизм — «generic prior reset» ближе к ходу.
- Содержимое: однострочное напоминание идентичности (кто агент, ключевой регистр/ценности) + один пример ответа «в характере» (one-shot demo). Сгенерировано из identity-доков, закэшировано.
- Идемпотентность: не более одного A-anchor на ход. Инъекция per-turn (не персистится в историю сессии — синтетический ход только в собранном `messages`, в БД не пишется).

## 5. Конфигурация

`[agent.drift]` в `agent.toml` — по образцу `[agent.soul]` (поле на `AgentSettings`, `#[serde(default)]`):

```toml
[agent.drift]
enabled = false        # opt-in, дефолт выключено
threshold = 0.05       # порог drift_score для инъекции A-anchor
min_history = 6        # минимум сообщений в истории до детекции
```

Константы (не конфиг, YAGNI): число якорей (~10+10), сколько последних ответов эмбеддить (1–2), базовый негативный набор.

Валидация — в `AgentConfig::load()` (как `SoulConfig::validate`, этап A): `threshold ∈ [−2.0, 2.0]`, `min_history ∈ [2, 50]`.

## 6. Интеграция в context_builder

- Новый deps-метод (в `ContextBuilderDeps`, реализация в `engine/context_builder.rs`): `async fn drift_check(&self, history, session_id) -> DriftOutcome` где `DriftOutcome { score: Option<f32>, inject_a_anchor: Option<String> }`. `(None, None)` при `!drift.enabled`, пустом кэше или ошибке (fail-soft, warn).
- Вызов в `build()` ПОСЛЕ сборки `messages` (system + history), ДО финализации. При `inject_a_anchor` — вставить синтетический ход перед последним user-сообщением.
- `ContextBreakdown` += поле `drift: usize` (размер A-anchor в chars/4, 0 если не вставлен) для наблюдаемости; `total()` += drift.
- Лог каждый ход: `drift_score`, `injected` (bool) — на уровне info/debug, чтобы наблюдать детектор на живом трафике.

## 7. Наблюдаемость (важно — доказательная база детекции слабая)

Метод детекции (Compass AUC 0.83) — из одноавторного препринта без независимой репликации (находка 8, оговорка). Поэтому:

- **drift_score логируется каждый ход** (даже когда ниже порога) — оператор видит поведение детектора на реальном трафике.
- Срабатывания A-anchor считаются (метрика/лог).
- Порог конфигурируемый, старт консервативный.
- **v1: только логи** (drift_score + injected каждый ход) + `ContextBreakdown.drift`. Status/regenerate-эндпоинты — B2 (YAGNI для v1: наблюдение через логи достаточно, ручная регенерация = удалить файл).

## 8. Обработка ошибок (fail-soft, как весь soul-слой)

- Embedding последнего ответа упал → детекция пропущена, A-anchor не вставлен, ход идёт как обычно.
- Кэш якорей пуст (toolgate был недоступен) → пропуск + ленивый ретрай.
- `DRIFT_ANCHORS.md` отсутствует и генерация упала → детекция off + warn (ход не страдает).
- `enabled=false` → ни файла, ни embedding, ни вставки — **ноль изменений поведения** (регрессионный инвариант).

## 9. Тестирование

**Unit (Rust, без БД/сети):**

- `drift_score`: центроиды, косинусы, знак (recent ближе к negative → положительный скор; ближе к positive → отрицательный); краевые (пустые якоря, один якорь, нормализация нулевого вектора).
- Решение об инъекции: порог, min_history гейт.
- Парсинг `DRIFT_ANCHORS.md` (3 секции, извлечение A-anchor).
- Рендер/вставка A-anchor в правильную позицию `messages` (перед последним user).
- `ContextBreakdown.drift` в `total()`.

**Интеграционные (по возможности локально / сервер):**

- Ленивый эмбеддинг + кэш + инвалидация по mtime (с fake-эмбеддером).
- Skip-лист: `DRIFT_ANCHORS.md` не попадает в workspace-промпт.

**E2E (сервер, вручную):**

1. Включить `[agent.drift]` на агенте → первое использование генерирует `DRIFT_ANCHORS.md` (осмысленные якоря).
2. Индуцировать дрифт: длинный диалог, где «собеседник» навязывает чужой регистр/персону → drift_score растёт → при превышении порога A-anchor вставляется (виден в context-breakdown.drift + лог).
3. Регрессия: `enabled=false` → нет файла, нет embedding, промпт байт-в-байт как без drift.
4. Fail-soft: остановить toolgate → детекция пропущена, ход работает.

## 10. План внедрения (эскиз для writing-plans)

1. `[agent.drift]` DriftConfig + валидация в load.
2. Модуль `agent/drift/`: парсер DRIFT_ANCHORS.md, генератор якорей (LLM, разово), drift_score (чистая функция), A-anchor-рендер.
3. Кэш `CachedAnchors` на AgentConfig + ленивый embed_batch + mtime-инвалидация.
4. Skip-листы workspace (оба загрузчика) + шаблон/генерация файла.
5. context_builder: `drift_check` deps-метод + вставка A-anchor + `ContextBreakdown.drift` + логи.
6. Деплой (push + server-deploy.sh; UI не нужен — v1 без UI) + E2E на одном агенте, наблюдение drift_score на живом трафике неделю → тюнинг порога.

## Зависимости и связь с другими этапами

- **Использует (этап A, задеплоен):** SELF.md/IDENTITY.md/SOUL.md как источник для генерации якорей и A-anchor; toolgate embeddings; context_builder; паттерн `[agent.soul]` конфига + AgentConfig-поля; `ContextBreakdown`.
- **Кандидаты на B2 (после наблюдения v1):** долгосрочный SELF.md-дрифт (heartbeat-пробы), ECP, weighted-top-k вместо центроида, trust-взвешивание.
- **Этап C (инициатива)** — независим, отдельная спека.
