# Программа стабилизации OPEX — дизайн (Фаза 0: Аудит)

**Дата:** 2026-07-08
**Статус:** дизайн одобрен, ждёт ревью спека → writing-plans
**Тип:** превентивное системное укрепление (не триаж инцидентов)

## 1. Цель и контекст

OPEX прошёл период очень быстрой разработки (562 коммита за 14 дней; крупные
фичи — file handlers, video-обработчики, handler valves — выкатывались
буквально накануне). Острого пожара в проде нет, но накопленный риск после
такого темпа надо системно выжечь **до** того, как он проявится. Это совпадает
с заявленной миссией проекта (Core Value: «найти и устранить баги,
несостыковки API, уязвимости и мёртвый код до того, как они проявятся в
продакшене»).

**Сигналы риска, уже видимые с порога:**

- ~2000 `unwrap()/expect()/panic!` в не-тестовом Rust-коде (перекликается с
  известным классом падений — срез `&str` по байт-offset → сессия `failed`).
- История содержит повторяющиеся классы дефектов: session/chat IDOR, SSRF
  (channel_action / browser / YAML-тулы), гонки жизненного цикла сессий
  (`guard_dropped`, ложный `failed` при отмене), `reqwest` без таймаута,
  deploy-швы (UI/миграции не авто-синкаются).

## 2. Периметр и режим (решения пользователя)

- **Драйвер:** превентивное укрепление.
- **Периметр:** вся система — Rust core, toolgate (Python), channels (TS),
  UI (Next.js) + инфра/деплой.
- **Режим:** audit-first — многоагентный аудит → ранжированный backlog →
  пользователь приоритизирует → чиним волнами с верификацией.
- **Подход к аудиту:** A (многоагентный Workflow: сетка «слой × ось» +
  адверсариальная верификация каждой находки), засеянный дешёвыми сигналами C
  (инструменты: clippy / cargo audit / grep panic-sites / tsc / npm audit /
  gen-types drift / выборка failed-сессий).

## 3. Декомпозиция (программа, не один план)

Укрепление всей системы по всем осям — это программа из нескольких волн:

- **Фаза 0 — Аудит** (проектируется здесь, запускается первой): многоагентный
  прогон → ранжированный backlog находок. Read-only, ничего не меняет в коде.
- **Фазы 1..N — Волны фиксов**: каждая волна = отдельный план (writing-plans),
  берётся из приоритизированного пользователем backlog, чинится по TDD,
  верифицируется, батчится, деплоится с подтверждения пользователя.

Каждая волна фиксов получает свой спек/план уже после того, как пользователь
увидит backlog и расставит приоритеты. Этот документ детально проектирует
**Фазу 0** и фиксирует **процесс волн** (раздел 8) как рамку.

## 4. Оси аудита

Восемь классов нестабильности, заземлённых на реальные болевые точки OPEX.

| # | Ось | Что ловим |
|---|-----|-----------|
| 1 | **Crash-surface / паники** | `unwrap/expect/panic!`, индексация, срез `&str` по байт-offset, переполнения; unhandled exceptions в TS/Python |
| 2 | **Error-handling & silent failures** | проглоченные ошибки (`let _ =`, `.ok()`, пустой catch), маскирующие фолбэки, фоновые таски, что молча умирают |
| 3 | **Concurrency, гонки, утечки** | TOCTOU (session-claim гонка), scope блокировок, `reqwest` без таймаута, утечки тасков/хендлов, корректность отмены |
| 4 | **Security — authz/изоляция** | IDOR/scope (session/chat/agent_id), SSRF (channel_action, browser, YAML-тулы), утечка секретов (vault-ключ в env субпроцесса), инъекции (SQL/path-traversal/provenance) |
| 5 | **API-контракты & целостность данных** | дрейф SSE-типов (Rust ↔ `sse-events.ts`), gen-types drift, безопасность миграций (необратимые/деструктивные), serde-rename рассинхрон |
| 6 | **Resource exhaustion / DoS-стойкость** | безлимитный ввод (upload/body/tool-output), корректность rate-limit, раздувание памяти, runaway-loop (корректность LoopDetector) |
| 7 | **Межслойные швы** | channel-WS session-корректность (drop сообщения при обработке), core↔toolgate (multipart/bytes, job-token HMAC), deploy-швы |
| 8 | **Мёртвый код & config-дрейф** | неиспользуемые модули/хендлеры, deprecated-таблицы, что ещё запрашиваются, мёртвые config-ключи, нарушения `.env`-политики |

Приоритет осей: 1–4 (реальные падения/безопасность) → 5–7 (корректность/швы)
→ 8 (гигиена).

## 5. Карта периметра → сетка finder'ов

Не полный декартов квадрат (слой×ось), а **курируемый набор ячеек**
(~20–30 finder-агентов): не каждая ось осмысленна для каждого слоя (напр.
char-boundary-паники — только Rust; SSRF — Rust core + toolgate).

- **Rust core** — `agent/pipeline` (execute/bootstrap/finalize/behaviour),
  `agent/providers`, `gateway/handlers` (chat/sse, channel_ws, files, sessions,
  services), `tools` (yaml/ssrf/system), `memory`, `secrets`, `db`,
  `process_manager`, `containers`, `workspace`
- **Rust-сателлиты** — `opex-watchdog`, `opex-memory-worker`, `opex-types`,
  `opex-db`
- **toolgate (Python)** — handlers (builtin+workspace), ctx API, `normalize`,
  providers, job-runner, endpoints
- **channels (TS)** — адаптеры, WS-клиент, upload, handshake
- **UI (Next.js)** — `chat-store` + `streaming-renderer` (реконнект),
  SSE-парсинг, auth, API-слой
- **Инфра/deploy/DB** — `migrations/`, `server-deploy.sh` + `deploy-ui.sh`,
  systemd, `docker/`, `opex.toml`

Точный список ячеек генерируется на старте Фазы 0 из этой карты + seed-сигналов
(раздел 6) и фиксируется в плане.

## 6. Структура Workflow (Фаза 0, read-only)

```text
Stage 0 — Seed-сигналы (дёшево, детерминированно):
  clippy -D warnings · cargo audit · grep panic-sites/unwrap ·
  tsc · npm audit · gen-types drift · выборка failed/guard_dropped
  сессий из session_timeline (на проде, если дотянемся)
        ↓ «leads»-бандл раздаётся finder'ам
Stage 1 — Finders (параллельно, по ячейкам, pipeline не barrier):
  каждый → структурные находки {file,line,axis,severity,
  confidence,blast_radius,failure_scenario,evidence}
        ↓ находка утекает в верификацию, как только finder закончил
Stage 2 — Адверсариальная верификация (на каждую находку):
  N=3 агента-скептика по умолчанию (для S0/S1 можно поднять),
  каждый с ЛИНЗОЙ (воспроизводится? путь достижим? уже защищено?)
  → большинство «refute» убивает находку → CONFIRMED vs PLAUSIBLE
Stage 3 — Дедуп + ранжирование (barrier):
  межслойный дедуп → сортировка severity×confidence×blast
Stage 4 — Completeness-критик:
  «что не покрыто, какой шов не проверен?» → добор finder'ов
```

**Замечания по инструментам-сигналам (Stage 0):**

- `cargo test`/`clippy`/`make test-db` — авторитет на **сервере**; локальный
  Windows Rust-тесты стабильно не гоняет. Seed-инструменты, что можно
  прогнать локально (grep, tsc, npm audit), запускаем локально; Rust-сигналы —
  на сервере или помечаем как «прогнать на сервере».
- `session_timeline` failed/`guard_dropped` выборка — только если дотянемся до
  прода; иначе Stage 0 работает без телеметрии (не блокирует аудит).

Конкурентность ограничена рантаймом Workflow (~min(16, cores−2)); общий лимит
агентов — рантаймовый backstop.

## 7. Рубрика severity + формат backlog

- **Severity:** `S0` reachable-в-проде crash/data-loss/security ·
  `S1` деградация стабильности (hang/leak/silent) при реалистичных условиях ·
  `S2` корректность/контракт с ограниченным радиусом ·
  `S3` мёртвый код/гигиена/латентное.
- **Confidence:** `CONFIRMED` (проверено) vs `PLAUSIBLE` (пережило верификацию,
  но не доказано).
- **Blast:** whole-system / subsystem / single-flow.
- **Артефакт:** `docs/audits/2026-07-08-stability-audit-findings.md` —
  ранжированная таблица + детали на каждую находку (id, ось, слой, `file:line`,
  severity, confidence, blast, сценарий отказа, доказательство, эскиз фикса).
  Опционально дублируется **Artifact-страницей** (hosted) для удобной
  приоритизации. Именно этот backlog пользователь размечает на волны.

## 8. Процесс фикс-волн (Фазы 1..N)

На каждую одобренную волну (батч находок, гринлайтнутый пользователем):

- `writing-plans` → per-finding TDD (тест, воспроизводящий баг — где реально;
  часть проверяется только E2E, помечается явно).
- Правки **в master** (предпочтение пользователя: работа в master).
- Верификация: CI-тройка (`cargo test --workspace` + `tsc` + gen-types drift) ·
  `clippy -D warnings` · авторитет `make test-db` **на сервере** ·
  toolgate pytest · vitest из `ui/`.
- Батчим → **подтверждение на push/deploy у пользователя** →
  `make remote-deploy` (сборка на сервере) → E2E на проде.
- Без Co-Authored-By в коммитах; инкрементальный деплой (не переустановка).

## 9. Ограничения (из истории проекта)

- Rust + rustls-tls only, никакого OpenSSL.
- Deploy-таргет: x86_64 сервер `aronmav@188.246.224.118`, сборка на сервере
  (`~/opex-src`, `make remote-deploy`). Pi выведен.
- Исправления не должны ломать API-контракты или миграции (backward compat).
- Никаких push/деплоев без явного подтверждения пользователя.
- CI-тройка обязательна перед push (server `make test-db` — только opex-core;
  CI гоняет `cargo test --workspace` + tsc + gen-types drift).
- Vitest гонять только из `ui/`; локальный Windows не гоняет Rust-тесты.
- Осторожно с `scp` в `~/opex-src` без commit — блокирует `git pull --ff-only`.

## 10. Non-goals / YAGNI

- **Не** добавляем новые фичи в рамках этой программы (feature-freeze по духу).
- **Не** делаем несвязанный рефакторинг ради красоты — только то, что снижает
  риск нестабильности или необходимо для конкретного фикса.
- **Не** переписываем подсистемы целиком; предпочитаем точечные,
  верифицируемые изменения.
- **Не** трогаем то, что история уже пометила как «не повторять» (напр.
  откачённые map-reduce/checklist в видео-пайплайне).

## 11. Критерий готовности Фазы 0

Фаза 0 считается завершённой, когда:

1. Прогнан многоагентный аудит по всем ячейкам сетки (раздел 5) со Stage 0–4.
2. Каждая находка в backlog имеет verdict (CONFIRMED/PLAUSIBLE) и заполненную
   рубрику (severity/confidence/blast/сценарий/доказательство).
3. Backlog записан в `docs/audits/2026-07-08-stability-audit-findings.md`
   (+ опц. Artifact) и передан пользователю на приоритизацию.
4. Completeness-критик (Stage 4) не выявляет непокрытых ячеек/швов, либо
   выявленные добраны отдельным раундом finder'ов.
