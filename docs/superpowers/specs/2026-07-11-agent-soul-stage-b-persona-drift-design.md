**Статус:** ждёт финального ревью оператора
**Программа:** этап B из трёх («душа агента»). Фундамент (этап A+D) задеплоен 2026-07-10.
**Исследовательская база:** [docs/research/2026-07-09-agent-soul-research.md](../../research/2026-07-09-agent-soul-research.md), находки 6–9.
**Опирается на:** [foundation-спеку](2026-07-09-agent-soul-foundation-design.md) (этап A: паттерны config/AgentConfig-полей, context_builder, session_timeline).

## Цель

Персона-дрифт быстрый и повсеместный (LLaMA2 теряет персону за ~8 ходов; 17/23 фронтир-моделей дрифтуют; статичный SOUL.md/IDENTITY.md его НЕ держит — находки 6, 7). Этот этап добавляет **пассивный контур наблюдения за внутрисессионным дрифтом личности**: дешёвый embedding-only детектор, сравнивающий недавнее поведение агента с его же ранними ходами этой сессии.

### Решения, зафиксированные с оператором (ревизия 2 — по итогам ревью)

| Вопрос | Решение |
| --- | --- |
| Тип дрифта | Внутрисессионный (агент перенимает персону собеседника / теряет регистр за длинный диалог). |
| Метод детекции | **Self-baseline (SPASM-style):** `drift = 1 − cos(recent_turn, baseline_centroid)`, где baseline = собственные ранние ответы агента в ЭТОЙ сессии. Embedding-only, без LLM. **Ревью показало: исходный anchor-косинус доминируется темой, ложит false-positive на cold-start, а редактируемый файл якорей = security-дыра. Self-baseline чинит всё это разом.** |
| Объём v1 | **Только detect + структурированный лог** (в `session_timeline`). Коррекция (A-anchor) — фаза 2, после недели наблюдения детектора на живом трафике. Не корректируем по невалидированному детектору. |

### Почему self-baseline, а не anchor-косинус (обоснование из ревью)

- **Топик-конфаунд:** passage-эмбеддинги кодируют тему >> стиль. Эмбеддинг ответа vs идеализированные статичные якоря колеблется по ТЕМЕ, не по персоне. Self-baseline сравнивает ранние и поздние ходы ОДНОГО агента в ОДНОЙ сессии → тема общая, сигнал = именно смещение поведения.
- **Cold-start false-positive:** идеализированные prose-якоря ≠ реальному терсовому/код-выводу → in-character агент ложно «дрифтует». Self-baseline берёт эталоном РЕАЛЬНЫЙ ранний вывод агента.
- **Security:** редактируемый файл якорей → агент/инъекция отравляет детектор (fail-open) ИЛИ пишет произвольное в A-anchor → self-prompt-injection. Self-baseline не использует файл для детекции — отравить нечего.
- **Верность исследованию:** это и есть SPASM-формула (агент — свой baseline), просто baseline берётся из ответов, а не из ответов на пробы-вопросы (последнее стоило бы LLM-вызовов — от чего оператор отказался).

### Вне скоупа v1

- **Коррекция A-anchor** — фаза 2 (см. §10), со всеми security-требованиями из ревью.
- **Долгосрочный дрифт** (эволюция SELF.md за недели) — отдельный контур на heartbeat, кандидат на B-late.
- **ECP** — валидирован только на ≤25-ходовых симуляциях.
- Этап C (инициатива) — отдельная спека.

## 1. Архитектура

Пер-ходовой embedding-only детектор в `context_builder`, полностью opt-in через `[agent.drift]`. **В v1: ноль новых файлов, ноль инъекций в промпт, ноль LLM-вызовов, ноль новых таблиц.** Только эмбеддинги + запись метрики в существующий `session_timeline`. Fail-soft везде. При выключенном флаге — ноль изменений поведения.

Поток данных (каждый ход, при `enabled`):

1. Из истории отобрать **собственные** (текущего агента) ответы с натуральным текстом.
2. Если их `< baseline_turns + 1` — пропуск (baseline не установлен, детекции нет).
3. **baseline** = центроид эмбеддингов первых `baseline_turns` собственных ответов сессии (считается ОДИН раз, кэшируется пер-сессия).
4. **recent** = эмбеддинг последнего собственного ответа (+1 embedding/ход).
5. `drift_score = 1 − cos(recent, baseline_centroid)` ∈ [0, 2].
6. Записать структурированную метрику в `session_timeline` (score + компоненты). Если `score > threshold` — дополнительно warn-лог/метрика «drift detected». **Никакой инъекции в v1.**

## 2. Детектор (алгоритм)

Чистая функция (тестируемо без БД/сети):

```text
baseline_centroid = mean(normalize(embed(first_k_own_assistant_turns)))
recent            = normalize(embed(last_own_assistant_turn))
drift_score       = 1 − cos(recent, baseline_centroid)      # ∈ [0, 2]; выше = дальше от раннего себя
```

- **Что считается «собственным ответом»:** сообщение с `role=assistant` И `agent_id == текущий агент` (мультиагентная сессия: чужие/peer-ответы НЕ берутся — ревью F10) И непустым натуральным текстом (tool-call-only ответы без текста пропускаются — ревью F12).
- **baseline_turns** (дефолт 3): сколько первых собственных ответов формируют эталон. Ранние ходы наименее дрифтованы.
- **min_history-гейт:** детекция включается только когда собственных текстовых ответов ≥ `baseline_turns + 1` (есть эталон + есть с чем сравнить). На коротких сессиях — пропуск, экономим embedding.
- **Нормализация:** каждый эмбеддинг нормируется перед усреднением (равный вес якорям, без доминирования магнитуды); центроид не единичный — cos использует только направление, это корректно (ревью F4 подтвердило математику).
- **Краевые:** нулевой вектор (embedding вернул нули) → пропуск; один baseline-ход → центроид = он сам.

## 3. Кэш baseline (пер-сессия)

baseline считается один раз на сессию и переиспользуется каждый ход:

- Session-scoped кэш `DashMap<Uuid /*session_id*/, Arc<Vec<f32>> /*baseline_centroid*/>` в in-core состоянии (рядом с `session_pools`/`session_locks` этапа A). Заполняется при первой успешной установке baseline (когда набралось `baseline_turns` собственных ответов), переиспользуется. dashmap уже в дереве.
- **Эвикция (v1): soft-cap backstop** — при `len() ≥ MAX_BASELINES` (2000) удаляется одна запись перед вставкой. **НЕ эвиктим на finalize намеренно:** фундамент переиспользует `done`-сессии в 4-часовом окне для непрерывности чата — эвикция baseline на finalize(done) заставила бы переиспользованную сессию пере-установить baseline из уже удлинённой истории (другие «ранние ходы»), теряя непрерывность эталона. Backstop бесконечный рост памяти ограничивает, а живые/переиспользуемые сессии эталон сохраняют. (dashmap-эвикция: ключ извлечь в отдельный `let` до `remove` — Iter держит shard-guard, иначе self-deadlock.)
- Baseline НЕ инвалидируется в течение сессии (эталон — ранние ходы, они фиксированы). recent эмбеддится каждый ход (не кэшируется).
- **Оговорка (marginal):** `session_load_messages` обрезает историю до последних `max_history_messages` (дефолт 50). Если сессия обрезана ДО установки baseline, «первые baseline_turns» — самые ранние В ОКНЕ, не истинно первые ходы сессии. Кусается только при `max_history_messages < ~10` И включённом drift — вырожденная конфигурация, detect-only; принято.
- Toolgate недоступен при установке baseline → baseline не ставится, ретрай на следующем ходу (fail-soft).

## 4. Наблюдаемость — центральная часть v1

Смысл v1 — понять, ловит ли детектор реальный дрифт на живом трафике (метод — из препринта, доказательная база слабая, находка 8). Поэтому лог должен быть **декомпозирован и queryable**, а не один скаляр в grep-логи (ревью F13):

- Каждый ход при активной детекции — запись в `session_timeline` (существующая таблица, `event_type='drift_probe'`, payload JSONB): `drift_score`, `cos_recent_baseline`, `own_assistant_turns` (сколько), `baseline_turns_used`, `history_len`, `over_threshold` (bool). Таблица уже ephemeral+prune (этап A) — идеальный дом для недели наблюдения, без новых таблиц.
- Плюс info/debug-лог строкой (score + over_threshold) для быстрого глаза.
- Оператор за неделю смотрит распределение drift_score по сессиям/агентам, коррелирует с реальным ощущением дрифта, тюнит `threshold` перед фазой 2 (коррекцией).

## 5. Конфигурация

`[agent.drift]` в `agent.toml` — по образцу `[agent.soul]`:

```toml
[agent.drift]
enabled = false        # opt-in, дефолт выключено (в v1 = detect+log)
threshold = 0.15       # drift_score, выше которого ход помечается over_threshold (для лога/фазы 2)
min_history = 6        # минимум сообщений истории до попытки детекции
baseline_turns = 3     # сколько первых собственных ответов формируют baseline
# correct = false      # ЗАРЕЗЕРВИРОВАНО под фазу 2 (инъекция A-anchor); в v1 отсутствует/игнорируется
```

Валидация — в `AgentConfig::load()` (как `SoulConfig::validate`, этап A): `threshold ∈ [0.0, 2.0]`, `min_history ∈ [2, 50]`, `baseline_turns ∈ [1, 10]`. Порог 0.15 — стартовая догадка (реальный рабочий диапазон `1−cos` для связанного текста ~0.05–0.4, топик-зависим), тюнится на живых данных (ревью F6).

## 6. Интеграция в context_builder

- Новый deps-метод `async fn drift_probe(&self, history, session_id) -> ()` (в `ContextBuilderDeps`, реализация в `engine/context_builder.rs`). При `!drift.enabled` — ранний возврат. Иначе: отфильтровать собственные текстовые ответы, установить/взять baseline из кэша, заэмбедить recent через **`self.cfg().embedder`** (поле `Arc<dyn EmbeddingService>` на AgentConfig — ревью F1 подтвердило: НЕ `MemoryStore::embedder()`, тот недоступен через `MemoryService`-трейт; путь как у `select_top_k_tools_semantic`), посчитать score, записать в timeline. Fail-soft: любая ошибка → лог + возврат, ход не страдает.
- Вызов в `build()` после загрузки истории (детекция читает прошлые ответы — доступны). **Ничего не вставляется в `messages`** (детект-онли).
- `ContextBreakdown` НЕ меняется в v1 (нет инжектированного контента; наблюдаемость — через timeline, не через prompt-size breakdown).
- **Явно: детекция отсутствует на путях, минующих context_builder** — `run_subagent` и `/v1/chat/completions` (как soul-блоки этапа A). Чтобы будущий рефактор случайно не добавил детекцию/инъекцию на путь без обвязки (ревью #6).

## 7. Стоимость

- +1 embedding/ход (recent) после установки baseline; `baseline_turns` эмбеддингов один раз на сессию (кэш).
- **С этапом A суммарно 2 embedding/ход** (soul L1 эмбеддит текст user-сообщения, drift эмбеддит последний ответ — РАЗНЫЕ строки, не схлопываются в один вызов, но могут разделить один `embed_batch` round-trip к toolgate — ревью F11). v1: держим раздельно ради простоты; батчинг — кандидат-оптимизация. Только для opt-in агентов.
- `embed_batch` принимает `&[&str]` (не `&[String]`) — ревью F1.
- `tokio::fs`/off-path генерация — не актуально в v1 (нет файла, нет LLM).

## 8. Обработка ошибок (fail-soft, как весь soul-слой)

- Embedding упал → детекция пропущена, метрика не записана, ход идёт как обычно.
- Baseline не установлен (toolgate был недоступен) → пропуск + ретрай на следующем ходу.
- `session_timeline`-запись упала → warn, ход не страдает (метрика best-effort).
- `enabled=false` → ни embedding, ни записи — **ноль изменений поведения** (регрессионный инвариант).

## 9. Тестирование

**Unit (Rust, без БД/сети):**

- `drift_score`: нормализация + центроид + `1−cos`; recent==baseline → score≈0; recent ортогонален baseline → score≈1; краевые (один baseline-ход, нулевой вектор).
- Отбор собственных текстовых ответов: фильтр по `agent_id`, пропуск tool-only/пустых, мультиагент (чужие ответы не берутся).
- Гейты: `min_history`, `baseline_turns + 1`.
- Формирование timeline-payload (поля на месте).

**Интеграционные (сервер/по возможности локально с fake-эмбеддером):**

- baseline устанавливается один раз, кэшируется, переиспользуется; soft-cap эвикция при переполнении (без deadlock).
- timeline-запись `event_type='drift_probe'` с корректным payload.

**E2E (сервер, вручную):**

1. Включить `[agent.drift]` на агенте → короткая сессия: детекции нет (мало ходов).
2. Длинный диалог, где «собеседник» навязывает чужой регистр → drift_score в `session_timeline` растёт по ходам; при превышении порога — `over_threshold=true` + warn-лог.
3. Регрессия: `enabled=false` → нет embedding, нет timeline-записей, промпт байт-в-байт как без drift.
4. Fail-soft: остановить toolgate → детекция пропущена, ход работает.
5. Мультиагент: в session-pool сессии чужие ответы не влияют на baseline/recent агента.

## 10. Фаза 2 — коррекция A-anchor (документированный follow-up, НЕ в v1)

После недели наблюдения детектора и тюнинга порога — добавить автокоррекцию. **Все security-требования из ревью — обязательны:**

- **A-anchor-артефакт защищён как SELF.md:** если хранится в файле (`workspace/agents/{Name}/A_ANCHOR.md` или секция) — добавить в `IDENTITY_FILES` + `is_read_only` (write-protect для ВСЕХ агентов); операторские правки/регенерация — только через workspace-API/эндпоинт, не агентскими тулами. (Иначе — self-prompt-injection канал, как поймало security-ревью.)
- **Генерация из RENDER-формы SELF.md**, не из сырого файла: реюз `render_self_block` (этап A), чтобы не потерять render-time рамку/ре-сериализацию, на которых стоит принятие остаточного риска этапа A (ревью #3).
- **Скан на генерации:** прогон `content_security::scan_for_block` + `detect_prompt_injection`; совпадение → отклонить/регенерировать.
- **Инъекция — PREPEND в реальное user-сообщение**, НЕ отдельным синтетическим user-ходом (иначе два подряд user-хода → Gemini отвергает, Anthropic version-dependent — ревью F7 + сверка кода подтвердили). Или в `bootstrap` step 9 перед push реального user. Внутри явной рамки; one-shot demo помечен как пример.
- **Раздельный флаг** `correct` (Phase 2), **гистерезис/кулдаун** (не инжектить каждый ход подряд — ревью F16), `ContextBreakdown.drift` для наблюдаемости инъекции.
- Кандидаты на выбор порога/сигнала для коррекции: top-k max по негативам, calibration-гейт (ревью F4/F5) — если решим дополнить self-baseline anchor-сигналом.
