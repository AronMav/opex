## Phase 68 — Uploads to PostgreSQL + Provider Modularisation

Two themed waves on top of v0.27.0: agent icons / tool outputs / chat attachments migrate from the `workspace/uploads/` filesystem into a single polymorphic PostgreSQL `uploads` table (closing the long-standing "icons disappear on deploy" bug), and the OpenAI / Anthropic / Google provider adapters are split into focused sibling modules so each file fits in a single review window.

### Highlights

* **Polymorphic `uploads` table.** Migration `052_uploads_table.sql` introduces one table keyed by `id UUID` with `(owner_type, owner_id)` discriminator — three classes: `agent_icon` (one row per agent, never expires), `tool_output` (server-side TTS/imagegen, 30-day retention), `client_upload` (chat composer + channel bridge media, 30-day retention). 20 MB CHECK constraint matches handler-level caps with application-side defense-in-depth.
* **HMAC-signed read endpoint.** `GET /api/uploads/{id}?sig=&exp=` replaces `GET /uploads/{filename}` with id-based addressing and HKDF-derived signing reused from Phase 64 SEC-03. The route is in `PUBLIC_PREFIX` so `<img>` / `<audio>` tags work without bearer headers; security comes from the HMAC.
* **XSS hardening.** Write side rejects `text/html`, `image/svg+xml`, `application/javascript`, and anything outside the inlineable image/audio/video families (`image/svg+xml` is excluded — SVG can carry script). Read side always sets `X-Content-Type-Options: nosniff` and attaches `Content-Disposition: attachment; filename="{id}"` for any MIME outside the inlineable set.
* **Icons survive every deploy.** `Makefile::deploy` ships the binary + UI + migrations but never copied `workspace/uploads/` — every binary swap erased agent avatars. With the new table the data lives in Postgres; restart-cycle and full deploy were both exercised against the live Pi and icons survived intact.
* **Agent self-service.** New JSON endpoint `POST /api/agents/{name}/icon/json` accepts `{mime, data_base64}` and a YAML tool `workspace/tools/set_my_icon.yaml` lets base agents set their own avatar (workflow: `generate_image` → base64 → `set_my_icon`). The multipart `PUT /api/agents/{name}/icon` is the UI path.
* **`AgentEditDialog` rewired.** UI no longer posts to `/api/media/upload` for icons — it sends the multipart `image` field directly to `PUT /api/agents/{name}/icon` and reads `icon_url` from the response. New agents see a toast asking to save first, mirroring the backend contract.
* **DTO factory batch-prefetch.** `signed_icon_url` switched from a per-agent DB lookup inside the DTO factory to a HashMap built once per handler via `db::uploads::list_agent_icon_ids`. Eliminates N+1 on `GET /api/agents`.
* **Hourly cleanup cron.** New `Scheduler::add_uploads_cleanup_hourly` reaps expired `tool_output` / `client_upload` rows; `agent_icon` rows have `expires_at = NULL` and are immune.
* **Rename / delete migrate icon rows.** `rename_agent_id_in_tables` updates `uploads.owner_id` for `agent_icon` rows; `cleanup_agent_data` deletes them. Without this the partial unique index orphaned old rows or leaked permanent rows on every delete cycle.

### Provider modularisation (W1)

* **`crates/hydeclaw-core/src/agent/providers/openai/`** is now a directory: `mod.rs` (entry), `request.rs` (body builder), `response.rs` (non-streaming response types), `stream.rs` (SSE streaming), `minimax_xml.rs` (XML tool-call parsing). Each file fits in a single LLM context window.
* **`agent/providers/google/`** mirrors the same shape: `mod.rs`, `request.rs`, `response.rs`.
* **Visibility tightened** after the split — internal helpers downgraded from `pub` to `pub(super)` / file-local so the provider tree no longer leaks implementation details into the rest of the crate.
* **Acceptance commit** locks the W1 baseline and is the cut point for future provider work.

### Reliability

* **Defense-in-depth size caps.** Migration CHECK `20 MB`, `db::uploads::MAX_UPLOAD_BYTES = 20 * 1024 * 1024`, handler-level caps (`10 MB` for icons, `20 MB` for media). Both DB-side and application-side rejections happen before the row is written, so a runaway YAML tool with a 50 MB binary response surfaces a clean error instead of a CHECK violation.
* **URL TTL aligned with retention.** Tool-output URLs were signed with `HISTORICAL_URL_TTL_SECS` (~50 years) while rows expired in 30 days — leaving a "URL valid, row gone" window of 49 years 11 months. URLs are now signed for `retention_days * 86_400`; row and URL expire together.
* **JSON body cap matches base64 overhead.** `POST /api/agents/{name}/icon/json` now layers `DefaultBodyLimit::max(MAX_BYTES * 4/3 + 4 KiB envelope slack)` — the handler-level "icon must be <= N bytes" error is reachable for the full 10 MB binary range instead of capping at ~7.5 MB (the previous shared body limit applied base64 inflation against the binary cap).
* **`/api/` prefix preserved in localhost downloader.** Voice-message STT, image vision, and the `analyze_image` YAML tool extract the path segment via `find()` to re-prepend `http://localhost:18789`. After the wave `find("/uploads/")` was matching the inner `/uploads/` inside `/api/uploads/...` and stripping `/api`, breaking all three flows silently. Fixed by preferring the longer prefix.

### Database

* `052_uploads_table.sql` — `uploads` table, three indexes (owner lookup, expires-at partial, agent-icon partial unique), 20 MB CHECK.
* No backwards-incompatible schema changes outside the new table.
* Old `AgentSettings.icon` TOML lines parse silently (no `deny_unknown_fields`) — Pi sweep via `sed -i '/^icon = /d'` performed in-place; release tarball ships no `icon = …` lines in scaffold or fixtures.

### Operational notes

* **Pi upgrade is a normal `update.sh`** — the new migration applies on startup, `workspace/uploads/` is no longer touched. Existing filesystem icons are left in place (un-served) until manual cleanup; re-upload via UI or `set_my_icon` to populate the DB.
* **`POST /api/media/upload`** preserved its response shape `{url, filename, size}` (filename = UUID string) so `ChatComposer` and `channels/src/bridge.ts` compile unchanged.
* **`[uploads] signed_url_ttl_secs`** scope clarified in `UploadsConfig` doc-comment: governs only `/api/media/upload` (client_upload) and workspace-files. `agent_icon` uses 50-year TTL (row is permanent), `tool_output` derives TTL from `cleanup.uploads_retention_days * 86_400`.
* **`[cleanup] uploads_retention_days = 30`** is the new tunable. Operator-tunable from `hydeclaw.toml`.

### Install / upgrade

Download the release tarball from this page:

* `hydeclaw-v0.29.0.tar.gz` — universal multi-arch tarball that includes `hydeclaw-core-aarch64`, `hydeclaw-core-x86_64`, the pre-built UI, channel adapters, toolgate sources, default config, and migrations. The `setup.sh` installer picks the right binary for your host architecture.

Fresh install:

```bash
mkdir -p ~/hydeclaw && cd ~/hydeclaw
curl -L -o hydeclaw-v0.29.0.tar.gz \
  https://github.com/AronMav/hydeclaw/releases/download/v0.29.0/hydeclaw-v0.29.0.tar.gz
tar xzf hydeclaw-v0.29.0.tar.gz
./setup.sh
```

Upgrade in place:

```bash
~/hydeclaw/update.sh hydeclaw-v0.29.0.tar.gz
```

Migrations apply automatically on startup. The new `uploads` table is created; agent icons begin landing there as soon as you re-upload them via the UI or `set_my_icon`.
