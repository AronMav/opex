# Коррекция искажённых названий в авто-транскриптах (term_fixer)

**Дата:** 2026-07-10 (rev2 — после трёхстороннего ревью: соответствие коду, логика, безопасность)
**Статус:** дизайн утверждён, реализация не начата

## Проблема

STT искажает названия продуктов, брендов и терминов: «T-G37» вместо «Tape J-37»,
«амбассадор» вместо «MBassador», «тейп-обработка» вместо «Tape». Ошибки попадают
и в сырой транскрипт (`transcribe`), и в конспект (`summarize_video`), делая
названия непригодными для поиска и покупки упомянутых инструментов.

Метод исправления (проверен вручную):

1. **Контекст** — рядом с кривым именем в транскрипте описано, ЧТО объект делает
   («тейп-обработка от Arturia — дисторшн, плёнка плавает»).
2. **Фонетика + числа** — «тейп» ≈ Tape, «амбассадор» ≈ MBassador, общий код
   «37» ↔ «J-37».
3. **Веб-поиск по описанию**, а не по кривому имени: «Arturia tape plugin
   distortion flutter» → Arturia Tape J-37.
4. **Сверка с реальностью** — продукт существует и совпадает по функции.

## Решение

Новый этап «detect → search → verify → apply» между STT и выдачей/конспектом.
Выполняется **один раз над транскриптом** — исправленные названия автоматически
попадают и в конспект, и во вложенный в заметку транскрипт.

Уверенные исправления подставляются молча; неуверенные помечаются в тексте
(«T-G37 *(вероятно Tape J-37?)*»). Все замены перечисляются в глоссарии.
Дополнительно сводка исправлений передаётся в digest-промпты (см. ниже) —
inline-пометки сами по себе не переживают map-reduce.

**Вне скоупа:** проверка, установлены ли найденные плагины у пользователя
(обработчик toolgate не имеет доступа к ПК пользователя); vision-обогащение;
исправление обычных слов (не названий).

## Архитектура

### Модуль `toolgate/term_fixer.py`

Новый модуль на корне toolgate (рядом с `video_helpers.py`).

⚠️ **Импорт:** НЕ копировать `parents[3]`-шов из обработчиков — он содержит
скрытый off-by-one (указывает на родителя toolgate, а не его корень) и работает
только потому, что async-runner (`handlers/runner.py`, `parents[1]`) сам
добавляет корень toolgate в `sys.path`. `term_fixer` импортируется простым
`from term_fixer import fix_terms` — оба обработчика async и всегда запускаются
через runner; тесты гоняются из toolgate.

```python
@dataclass
class Replacement:
    heard: str            # форма из транскрипта (из detect)
    variants: list[str]   # словоформы из текста (отфильтрованные, см. капы)
    corrected: str        # реальное название (из verify, санитизировано)
    confidence: str       # "high" | "low" (всё прочее трактуется как "low")
    description: str      # ТОЛЬКО из detect (из транскрипта), никогда из сниппетов

@dataclass
class FixResult:
    transcript: str            # исправленный текст (или исходный, если замен нет)
    replacements: list[Replacement]
    glossary_md: str           # готовый markdown-блок; "" если замен нет
    term_notes: str            # сводка исправлений для digest-промптов; "" если нет

async def fix_terms(ctx, transcript: str, language: str = "ru") -> FixResult
```

### Пороги и капы (константы модуля)

| Константа | Значение | Назначение |
| --- | --- | --- |
| `MIN_FIX_CHARS` | 300 | Транскрипт короче (после strip таймкодов) → этап пропускается БЕЗ единого LLM-вызова. Короткие голосовые не замедляются и не порождают внешних поисков. |
| `MAX_CANDIDATES` | 8 | Кап кандидатов суммарно (после dedup). Усечение логируется: `ctx.log.warning("term_fixer: %d candidates over cap, dropped: %s", ...)`. |
| `MAX_VARIANTS` | 10 | Кап словоформ на кандидата. |
| `MIN_VARIANT_LEN` | 3 | Словоформы короче — отбрасываются. |
| `MAX_CORRECTED_LEN` | 80 | `corrected` длиннее — кандидат отбрасывается (не обрезается). |
| `MAX_QUERY_LEN` | 200 | Поисковый запрос длиннее / многострочный — кандидат отбрасывается. |
| `DETECT_WINDOW_MIN` | 45 | Окно detect-чанкинга (минуты), переиспользует `split_transcript_by_time`. |
| `SEARCH_CONCURRENCY` | 4 | Параллельность поисков (`asyncio.gather` + `Semaphore`). |

### Шаг 0 — Пре-чеки

1. Valve `fix_terms` выключен → возврат исходника.
2. `len(strip_transcript_timecodes(transcript).strip()) < MIN_FIX_CHARS` → возврат исходника.
3. `not await ctx.has_capability("websearch")` → `ctx.log.warning`, возврат
   исходника. Без «сверки с реальностью» подстановки не делаем — LLM-догадки
   без поиска опаснее исходных ошибок. **Требует расширения ctx** (см.
   «Изменение контракта ctx»).

### Шаг 1 — Detect (LLM, по чанкам)

Транскрипт режется на окна по `DETECT_WINDOW_MIN` минут переиспользованием
`split_transcript_by_time` (перенести/импортировать из summarize_video; для
транскриптов без таймкодов — fallback: окна по ~24 000 символов). Короткий
транскрипт = одно окно = один вызов. **Полный транскрипт длинной лекции никогда
не уходит в один LLM-вызов** — map-reduce в digest существует именно потому,
что он не влезает; detect обязан следовать той же дисциплине (иначе на целевом
сценарии — длинные туториалы — этап молча отваливается по контексту или
таймауту 120с `ctx.llm.complete`).

Промпт (на окно): «Ниже фрагмент авто-транскрипта. Найди названия
продуктов/брендов/моделей/терминов, которые выглядят как фонетическое искажение
(несуществующие названия, кириллица там, где ожидается латинский бренд,
странные буквенно-цифровые коды). Для каждого верни JSON-объект: `heard`
(точная форма из текста), `variants` (все словоформы из текста), `description`
(что объект делает, по контексту), `query` (поисковый запрос на английском ПО
ОПИСАНИЮ, а не по кривому имени). Верни JSON-массив; если искажений нет —
пустой массив `[]`.»

Постобработка:

- **Парсер JSON:** снять markdown-заборы → `json.loads` → если объект с ключом
  `candidates`, взять его → если не распарсилось, извлечь первый `[...]`-блок
  regex-ом → иначе warning + окно даёт ноль кандидатов (не роняет остальные окна).
- Элементы без обязательных ключей / с не-строковыми значениями — отбрасываются
  поштучно.
- **Фильтр variants:** отбрасываются пустые/пробельные (пустой паттерн в
  `re.sub` вставил бы `corrected` между каждым символом — уничтожение
  транскрипта), короче `MIN_VARIANT_LEN`, чисто цифровые и цифро-дефисные
  (variant «37» изувечил бы таймкоды `[12:37]`, ломая `_parse_line_minute`,
  чанкинг и заметку). Кандидат без выживших variants отбрасывается.
- **Dedup между окнами** по нормализованному `heard` (casefold), затем кап
  `MAX_CANDIDATES` с warning-ом об усечении.
- **Фильтр query:** одна строка, ≤ `MAX_QUERY_LEN`, иначе кандидат отброшен
  (detect-LLM обрабатывает недоверенный текст и полностью контролирует query,
  уходящий внешнему провайдеру).
- Каждому выжившему кандидату присваивается стабильный `id` (индекс) — verify
  обязан ссылаться на него.

### Шаг 2 — Search

Для каждого кандидата один `ctx.search(query, max_results=5)`; поиски идут
параллельно (`asyncio.gather` + `Semaphore(SEARCH_CONCURRENCY)`). К кандидату
прикладываются `title` + `content` + `url` результатов — websearch-провайдеры
нормализуют выдачу именно в эти три ключа (`content`, НЕ `snippet`; Brave кладёт
`description` в `content`). Не-списочный/мусорный результат нормализуется через
`.get(..., "")`, элементы без `title`+`content` отбрасываются. Ошибка одного
поиска → кандидат уходит в verify с пометкой «без результатов».

### Шаг 3 — Verify (1 батч-вызов LLM на все кандидаты)

Промпт: «Для каждого кандидата (по `id`): услышано X, описание Y, выдача
поиска Z. Реши, какое реальное название имелось в виду. Критерии: фонетическое
сходство (тейп≈Tape, амбассадор≈MBassador), общие цифры/коды (37↔J-37),
совпадение бренда И функции с описанием из контекста. Верни для каждого `id`
один из исходов: `{"id": N, "corrected": "...", "confidence": "high"|"low"}`;
`{"id": N, "already_correct": true}` — услышанное само является реальным
названием (замен нет, в глоссарий не попадает); `{"id": N, "corrected": null}`
— подходящего продукта в выдаче нет (кандидат отбрасывается). `confidence:
high` только когда сходятся и фонетика, и функция; сомнение — `low`.»

Постобработка (санитизация — сниппеты недоверенные, SEO-отравленная выдача
может продиктовать произвольный `corrected`):

- Ответ join-ится с кандидатами **по `id`**; `corrected` для неизвестного `id`
  игнорируется (защита от кросс-кандидатной инъекции в батче).
- `corrected`: строка, одна строка (без `\n`), ≤ `MAX_CORRECTED_LEN`, длина не
  более 3× `len(heard)`, допустимый алфавит — буквы/цифры/пробел/`.-_&+()`
  (без markdown-символов `#[]!`, backtick, `<>`). Любое нарушение → кандидат
  **отбрасывается** (не обрезается). Не-строковый `corrected` → кандидат
  отброшен, остальные продолжают обрабатываться.
- `confidence` не из {high, low} → трактуется как `low`.
- No-op guard: `corrected` == `heard` (casefold) → трактуется как
  `already_correct`.
- `description` в глоссарий берётся ТОЛЬКО из detect-выхода — никогда из
  verify-ответа или сниппетов.

### Шаг 4 — Apply (чистый Python, без LLM, однопроходный)

Все пары (variant → replacement) всех кандидатов собираются в **один
regex-альтернатор**: варианты отсортированы по длине (длинные раньше), каждый
обёрнут границами слова `(?<!\w)…(?!\w)` (`\w` в Python unicode-aware —
кириллицу покрывает), матчинг регистронезависимый (`re.IGNORECASE`). Один
`re.sub` с lookup-функцией по всему тексту.

Свойства этой схемы (каждое — тест):

- **Нет каскадов:** замены применяются только к исходному тексту — `corrected`
  одного кандидата не матчится variant-ом другого («Tape J-37 J-37» невозможен).
- **Нет порчи слов:** «тейп» не матчится внутри «тейповый» (границы слова).
- **Регистр:** «Амбассадор» в начале предложения и «амбассадор» в середине
  заменяются одинаково.
- **Таймкоды/URL:** защищены фильтром цифровых variants + границами слова.
- **«Первое вхождение» для low** определено естественно: первый матч
  альтернатора по тексту среди всех variants кандидата (минимальный индекс).
- **Атомарность:** apply работает над копией; любое исключение → возврат
  исходного текста целиком (никаких частично применённых замен).

Логика подстановки: `high` → все вхождения заменяются на `corrected`; `low` →
только первый матч аннотируется `<найденная форма> (вероятно {corrected}?)`
(остальные вхождения не трогаются — не заспамить длинный транскрипт).

### Глоссарий и term_notes

`glossary_md` (все значения — одна строка, markdown-спецсимволы `*_[]()!` и
backtick экранируются, значения в «ёлочках» — heard/variants из недоверенного
транскрипта, corrected из недоверенной выдачи):

```markdown
## Исправленные названия
- «амбассадор» → **MBassador** — суб-бас плагин
- «T-G37» → *вероятно* **Arturia Tape J-37** (не подтверждено)
```

`term_notes` — компактная текстовая сводка тех же замен для digest-промптов:
«В транскрипте уже исправлены названия: "MBassador" (было "амбассадор");
"T-G37" вероятно означает "Arturia Tape J-37" (не подтверждено — упоминай с
пометкой "вероятно")».

Локализация: заголовок глоссария, шаблон аннотации «(вероятно X?)» и
`term_notes` — по `language`: `ru` → русские строки, всё остальное (включая
`auto`) → английские («## Corrected names», «(likely X?)»).

### Стоимость этапа

2 LLM-вызова (detect-окна: по одному на 45 минут транскрипта + 1 verify) +
≤ `MAX_CANDIDATES` параллельных поисков. Пустой detect по всем окнам → verify
и search не выполняются. Транскрипт < `MIN_FIX_CHARS` → ноль вызовов.

## Изменение контракта ctx (отдельная задача)

В `toolgate/handlers/context.py` добавляется публичный probe:

```python
class HandlerContext:
    async def has_capability(self, capability: str) -> bool:
        """True когда активный провайдер capability настроен (без вызова)."""
```

Реализация — `await registry.aget_active(capability) is not None` (тот же
резолв, что в `_CapabilityWrapper._resolve`, но без исключения). Без него
«пропуск этапа без websearch-провайдера» нереализуем: `RuntimeError("no active
websearch provider")` возникает только при первом `ctx.search`, т.е. ПОСЛЕ
оплаченного detect-вызова.

## Интеграция

### Valve (оба обработчика, default on)

```xml
<field name="fix_terms" type="bool" default="true" label="Исправлять названия"
       description="Определять искажённые STT названия (бренды, плагины, термины) и исправлять их через веб-поиск. Транскрипты короче 300 символов пропускаются."/>
```

Чтение — дословно как `include_transcript` (с `.strip()`):
`str(ctx.config.get("fix_terms") or "true").strip().lower() not in ("false", "0", "no")`.
Никаких `params`-переопределений (YAGNI). Порог длины — жёсткая константа
`MIN_FIX_CHARS`, не valve.

### transcribe.py

После empty-guard (фаза `saving 90` сохраняется):

```python
glossary = ""
if fix_terms_enabled:
    await ctx.progress("fix_terms", 60)   # суб-прогресс внутри fix_terms: 60/70/80
    fx = await fix_terms(ctx, text, language)
    text, glossary = fx.transcript, fx.glossary_md
await ctx.progress("saving", 90)
return ctx.result.text(text + (f"\n\n---\n{glossary}" if glossary else ""))
```

Глоссарий отделяется `---`, чтобы читающий выдачу агент не считал его речью.
Docstring и `<description>` хэндлера дополняются: при `fix_terms=on` текст
пост-обработан (не строго verbatim). Провенанс сохраняется: `summary_text`
оборачивается `wrap_file_output(..., trust="untrusted")` в core как и раньше —
web-производный `corrected` оказывается внутри untrusted-обёртки.

### summarize_video.py

Сразу после empty-guard, до digest:

```python
term_notes = ""
glossary = ""
if fix_terms_enabled:
    await ctx.progress("fix_terms", 40)   # суб-прогресс 40/43/46
    fx = await fix_terms(ctx, transcript, language)
    transcript, glossary, term_notes = fx.transcript, fx.glossary_md, fx.term_notes
```

- Конспект строится по исправленному транскрипту; **дополнительно** `term_notes`
  передаётся новым опциональным параметром в `build_single_pass_messages`,
  `build_chunk_messages` и `build_reduce_messages` и дописывается к system-промпту
  («Известные исправления названий: … Используй исправленные написания; названия
  с пометкой "вероятно" упоминай с этой пометкой»). Без этого inline-пометка low
  видна только одному map-чанку, а digest-LLM не знает конвенцию «(вероятно X?)»
  и может её потерять или молча «подтвердить».
- `glossary_md` передаётся в `build_note` новым опциональным параметром
  `glossary: str = ""` и вставляется **после тела конспекта, перед свёрнутым
  транскриптом**.
- Вложенный транскрипт в заметке — исправленный; сырой STT-текст при high-заменах
  восстановим только через глоссарий (принято).

### Прогресс / UI

Новая фаза `fix_terms` с суб-прогрессом (detect → search → verify — три
инкремента pct внутри фазы; фаза может жить дольше транскрибации):

- `PHASES` в [ui/src/components/chat/VideoProgressIndicator.tsx](../../../ui/src/components/chat/VideoProgressIndicator.tsx)
  — запись `fix_terms: { emoji: "🔎", key: "chat.video_phase_fix_terms" }`.
- `ru.json`: «Проверяю названия…»; `en.json`: «Verifying names…».

Core/channels не меняются: фаза проходит сырым ключом через WS-событие
`file_job_progress`; неизвестные фазы старый UI показывает сырым текстом
(fail-soft уже встроен). `TranslationKey` выводится из `ru.json` — правки только
локалей достаточно.

## Безопасность (принятые риски и компенсации)

| Вектор | Оценка | Компенсация |
| --- | --- | --- |
| Prompt injection через транскрипт → эксфильтрация через query | Низкий: detect-вызов — raw-completion (`/api/llm/complete`) без агентского контекста; в промпте нет ничего, кроме самого транскрипта. | Кап и однострочность query. |
| Приватность: query из личной голосовухи уходит поисковому провайдеру | Принято (SearXNG self-hosted у оператора; приоритетный провайдер настраивается). | `MIN_FIX_CHARS` отсекает бытовые голосовые; valve отключаем. |
| Prompt injection через веб-сниппеты → произвольный `corrected` | Главный вектор: SEO-отравленная выдача могла бы вставить текст во все вхождения. | Санитизация corrected (кап 80, одна строка, allowlist-алфавит, ≤3× heard, отброс при нарушении), join по id, description только из detect. |
| Markdown-инъекция через глоссарий (чат-выдача + vault-заметка) | Реален: заметка уходит в `workspace/` → memory-watcher → shared-память агентов БЕЗ untrusted-обёртки (существующий архитектурный гэп, term_fixer добавляет в путь второй недоверенный источник — web). | Экранирование и однострочность всех значений глоссария; ограничения на corrected выше. Риск записан как принятый. |
| Порча данных заменой (таймкоды, URL, числа) | — | Границы слова, фильтр коротких/цифровых variants, однопроходность, атомарность. |
| Накрутка ресурсов потоком голосовых | Низкий: handler_jobs-воркер последовательный; `access.mode=restricted` (дефолт) не пропускает чужих. | `MIN_FIX_CHARS`; параллелизм поисков ограничен семафором. |

## Обработка ошибок

Базовый принцип: **коррекция никогда не роняет джобу** — любая ошибка внутри
`fix_terms` логируется warning-ом и возвращает исходный транскрипт без замен.

| Сбой | Поведение |
| --- | --- |
| Нет активного websearch-провайдера | `ctx.has_capability` в шаге 0 → скип целиком, ни одного LLM-вызова. |
| Detect-окно: HTTP-ошибка / таймаут / невалидный JSON | Warning, окно даёт ноль кандидатов; остальные окна обрабатываются. Все окна пусты → выход с исходником. |
| Ошибка одного `ctx.search` | Кандидат уходит в verify с пометкой «без результатов»; verify отбрасывает неподтверждённых. |
| Verify упал / вернул мусор целиком | Warning, исходный текст. |
| Verify: битый один кандидат (не-строка, нарушение санитизации, неизвестный id) | Кандидат отброшен, остальные продолжают. |
| LLM выдумал словоформу (не найдена в тексте) | Ноль замен по ней; кандидат без единого матча не попадает в глоссарий. |
| Исключение внутри apply | Атомарный откат — исходный текст целиком. |

## Тестирование (TDD — тесты первыми)

`toolgate/tests/`, по образцу существующих (`test_handlers_builtin.py` —
`_FakeRegistry` + `build_context`; `test_handlers_summarize_video.py` —
`AsyncMock` на `ctx.llm.complete`).

1. **Чистые функции:**
   - парсер detect-JSON: валидный массив / заборы / объект `{"candidates": []}` /
     trailing-текст с `[...]`-блоком / мусор;
   - фильтры: пустые и пробельные variants, короче 3, цифровые/цифро-дефисные,
     кап variants, кап кандидатов с warning, кап/однострочность query;
   - санитизация corrected: длина, `\n`, markdown-символы, не-строка, 3×-лимит,
     no-op → already_correct;
   - apply: границы слова («тейп» не трогает «тейповый»), регистронезависимость,
     однопроходность (corrected содержит variant другого кандидата — не каскадит),
     длинные→короткие, low аннотирует первый матч среди всех variants,
     сохранность таймкодов и URL, атомарный откат при исключении;
   - глоссарий: экранирование markdown-спецсимволов, локализация ru/en;
   - term_notes: сборка, локализация.
2. **Оркестратор `fix_terms`** со стабами `ctx.llm` / `ctx.search` /
   `ctx.has_capability`: happy-path; порог `MIN_FIX_CHARS` (ноль вызовов);
   нет websearch (ноль вызовов); пустой detect по всем окнам; detect по
   нескольким окнам с dedup по heard (casefold); join verify по id, чужой id
   игнорируется; поиск упал у одного кандидата.
3. **Интеграция обработчиков** со стабом `fix_terms`: valve on/off (паттерн с
   `.strip()`); глоссарий в выводе `transcribe` после `---`, фаза `saving 90`
   сохранена; в summarize_video — глоссарий в заметке до транскрипт-блока,
   `term_notes` дошёл до prompt-builders (single-pass и reduce).
4. **`ctx.has_capability`** — новый тест в тестах context: true/false по
   наличию активного провайдера, без вызова provider-метода.

Rust-тесты не затрагиваются. Python-тесты toolgate гоняются локально и на
сервере (venv).

## Деплой

Toolgate — нативный Python-процесс на сервере: scp изменённых `.py`
(`term_fixer.py`, `handlers/context.py`, `handlers/builtin/transcribe.py`,
`handlers/builtin/summarize_video.py`, тесты) +
`POST /api/services/toolgate/restart`. UI (новая фаза + локали) — по
существующему UI-деплой-пути (`deploy-ui.sh`).
