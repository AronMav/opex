# Дизайн: инструмент агента для навигации и поиска по ИТС 1С (its.1c.ru)

**Дата:** 2026-07-03
**Ревизия:** v2 (внесено критическое ревью — секции риска, assisted-login, сужение
скоупа, ужесточение сериализации)
**Статус:** дизайн утверждён, ожидает вычитки → план
**Автор:** brainstorming-сессия

## 1. Задача

Дать агентам OPEX возможность искать и читать информацию в платной документации
1С на `its.1c.ru`, ведя себя как обычный залогиненный пользователь (не как
свежий бот, которого блокирует антибот-защита). Доступ легальный — у оператора
есть оплаченная подписка (один аккаунт / один сеат).

**Референс-ссылка от оператора:** `its.1c.ru/db/v854doc#bookmark:adm:TI000001410`
(руководство администратора платформы 8.5.4) — используется как приёмочная
цель E2E.

### Сценарий (утверждён)

Гибрид **поиск + чтение** без массового краулинга: агент точечно вызывает
`its_search(query)` и `its_read(ref)`; вся «грязь» (логин, сессия, кэш,
анти-бот-вежливость) спрятана под капотом.

### Охват v1 (сужен после ревью — S6)

Приёмка v1 — **базы документации** (тип `db/<name>`, как `v854doc`): там отлаживаем
надёжно логин, поиск, извлечение. Остальные разделы ИТС (новости, методички,
справочники) — **best-effort** через тот же generic-драйвер, без гарантий парсинга
в v1. Архитектура заложена расширяемой (см. §3.2), но «весь ИТС отлажен» —
цель следующих итераций, не v1.

### Доступ (утверждён)

YAML-инструмент `its`, доступный **всем агентам**. Единственная платная сессия
защищается сериализацией на уровне сервиса, а не ограничением на агентов.

## 2. Принципы и границы

- **Вежливая автоматизация, не гонка вооружений.** Персистентная залогиненная
  сессия, человеческий темп, кэш, разумные лимиты. **Не** строим обход защиты:
  никаких ферм решения капч, ротации резидентных прокси, массового спуфинга
  фингерпринтов. Капча / 2FA / глухой блок → внятная ошибка оператору, не попытка
  сломать.
- **`browser-renderer` остаётся 100% универсальным.** Никакого 1С-специфичного
  кода и никаких кредов внутри него.
- **YAGNI.** Полноценный декларативный DSL для логин-флоу не делаем. Site-специфику
  держим как generic-драйвер + маленький конфиг-словарь на сайт; первый и
  единственный словарь — ИТС.
- **Backward compatibility.** Существующие эндпоинты `browser-renderer`
  (`/extract`, `/screenshot`, `/automation` без профиля) и YAML-tool `browser`
  работают без изменений.

### 2.1. Принятие риска (S3)

Автоматический доступ к ИТС может нарушать пользовательское соглашение 1С; 1С
активно защищает портал. Даже «вежливая» автоматизация при детекте может привести
к **блокировке платного аккаунта** и потере подписки. Это осознанный риск, который
**оператор принимает на себя**. Дизайн минимизирует вероятность детекта
(персистентная сессия, человеческий темп, кэш, консервативный перехват — §4.1), но
не может её обнулить. Инструмент держим для личного, низкообъёмного использования.

## 3. Архитектура: универсальные примитивы вниз, знание о сайте — вбок

```text
агент → YAML-tool `its`  →  toolgate /its/{search,read}      ← ЗНАНИЕ о 1С:
                                   │                            селекторы логина,
                                   │                            URL поиска, DOM
                                   │                            контента, парсинг,
                                   │                            креды, кэш
                                   │ generic /automation, profile="its"
                                   ▼
                            browser-renderer                  ← УНИВЕРСАЛЬНО:
                                   │                            профили + stealth
                                   │                            + примитивы
                                   │ launch_persistent_context(/profiles/its)
                                   ▼
                            login.1c.ru / its.1c.ru
```

### 3.1. Что получает `browser-renderer` — только универсальное

Две generic-фичи, полезные любому сайту с авторизацией; обе backward-compatible.

1. **Именованные персистентные профили.**
   - `create_session(profile="<name>")` → сессия в персистентном контексте
     `launch_persistent_context(user_data_dir=/profiles/<name>, ...)`. Куки живут
     между рестартами сервиса.
   - `create_session()` без `profile` → эфемерная сессия ровно как сейчас
     (`browser.new_page()`), TTL 5 мин.
   - **Два инстанса Chromium — ожидаемо (S9).** `launch_persistent_context`
     запускает отдельный браузер-процесс с профилем, он **не** порождается от
     общего `browser`. Значит в сервисе сосуществуют: общий эфемерный `browser`
     (для профильных сессий — без изменений) + по одному персистентному контексту
     на каждый именованный профиль. Это нормально.
   - **Singleton-контекст на профиль.** Chromium физически лочит `user-data-dir`,
     поэтому на один профиль — ровно один живой персистентный контекст (создаётся
     лениво, живёт всё время жизни сервиса; страницы recycled). Отсюда сериализация
     выпадает бесплатно.
   - Персистентный профиль **не** подчиняется 5-мин TTL эфемерных сессий и **не**
     должен голодать generic-пул `max_concurrent: 2` (S12) — учесть в реализации,
     чтобы долгоживущий контекст не занимал слот generic-очереди.

2. **Stealth-хардненинг** (через `add_init_script` при создании контекста):
   реалистичный десктопный Chrome UA, `Accept-Language: ru-RU,ru`, патч
   `navigator.webdriver`, вменяемый viewport, отключение автоматизационных
   признаков через launch args / init-script.

`browser-renderer` не знает ни про 1С, ни про креды: он выполняет generic-команды
(`type` в поле X значение Y, `wait` селектора Z, `content`). Значения (включая
пароль) приходят как обычные аргументы действий; он их не хранит и не логирует.

**Безопасность профилей.** Параметр `profile` **не выставляется** в агентском
YAML-tool `browser` — иначе агент смог бы `create_session(profile="its")` и
«оседлать» залогиненную 1С-сессию. Профили использует только внутренний
оркестратор toolgate. (Опционально: `browser-renderer` принимает `profile` только
с internal-заголовком/от доверенного источника — уточнить на этапе плана.)

### 3.2. Где живёт знание о 1С — оркестратор в toolgate

Всё 1С-специфичное — модуль `toolgate/its.py` + роуты `/its/search`, `/its/read`:

- знает селекторы формы `login.1c.ru`, URL/механику поиска ИТС, DOM-селекторы
  контента, извлечение → markdown, кэш;
- держит креды (из vault), детектит «разлогинено» и гоняет логин **внутри
  профиля `its`**;
- всю реальную работу браузера делегирует `browser-renderer` через generic
  `/automation` с `profile="its"`. **Второго браузера не заводим.**

Почему toolgate: это уже «хаб инструментов», Python (удобно парсить —
`selectolax`/BeautifulSoup), managed-процесс, запущен с `--workers 1
--loop asyncio` → единственный процесс, `asyncio.Lock` там сериализует запросы
всех агентов через единственную сессию.

Site-специфику структурируем как **generic-драйвер + конфиг-словарь на сайт**
(селекторы формы логина, URL поиска, селекторы контента, признак «разлогинено»).
Первый словарь — только ИТС.

### 3.3. Агентский инструмент

`workspace/tools/its.yaml` — обычный YAML-tool, endpoint на внутренний toolgate
(`is_internal_endpoint` → стандартный `http_client`, без SSRF-блокировки).
Действия:

- `its_search` — параметр `query` (и опционально `db` для сужения охвата, см. S6);
  возвращает структурный список результатов;
- `its_read` — параметр `ref` (полный URL или закладка); возвращает markdown.

(Реализуемо как один tool с параметром `action`, либо два отдельных tool —
уточнить на этапе плана; агентский контракт одинаков.)

## 4. Данные и потоки

### 4.1. Логин (ленивый, консервативный перехват — S1)

Триггерится только когда `search`/`read` упирается в разлогиненное состояние:

1. Оркестратор заходит на лёгкий authed-URL.
2. Детект «разлогинено» (URL содержит `login.1c.ru` / присутствует селектор формы
   логина).
3. Если разлогинено: заполнить логин → пароль → submit → дождаться возврата на
   `its.1c.ru`.
4. Куки автоматически сохраняются в персистентном профиле → следующие вызовы
   логин пропускают.

**Консервативный перехват (защита от пинг-понга с человеком, S1).** Этот аккаунт
параллельно использует живой оператор. Чтобы не устроить войну перелогинов
(частые логины = триггер антибота/блокировки):

- **Cooldown на перелогин.** Инструмент не логинится чаще, чем раз в N минут
  (конфигурируемо, дефолт ~5 мин). Если только что залогинились и снова выкинуло —
  это признак борьбы за сессию → **не перезахватываем в цикле**.
- **Отступление.** При интерстишле «вы вошли в другом месте» / повторном выкидывании
  инструмент **отступает** и возвращает агенту ошибку `its_busy` («сессия ИТС занята,
  вероятно используется человеком, попробуйте позже»), а не отвоёвывает сессию.
- Спайк (§12) проверит, сколько **параллельных** сессий реально разрешает подписка —
  если несколько, ограничение можно ослабить.

> **Дефолт-решение (вето, если не так):** приоритет у живого человека; агент
> уступает. Альтернатива — агрессивный перехват — отвергнута как рискованная для
> аккаунта.

### 4.2. Assisted login / посев куки (S2)

Капча / 2FA / одноразовое подтверждение при логине сделали бы инструмент
перманентно мёртвым (автомат их не проходит и не должен). Фолбэк для **разового
посева** персистентного профиля `its`:

- **Вариант 1 — headful ручной вход:** оператор один раз проходит логин в контексте
  профиля `its` (headful/screenshare-режим browser-renderer или локально с тем же
  `user-data-dir`); куки оседают в томе профиля, дальше автомат живёт на них.
- **Вариант 2 — импорт куки:** оператор экспортирует куки ИТС из своего браузера →
  кладём в профиль `its`.

Что именно реализуем, зависит от спайка (есть ли капча/2FA вообще). Минимум —
предусмотреть операторский путь «засеять/обновить куки профиля вручную», чтобы
любой hard-блок логина не убивал инструмент навсегда.

### 4.3. `its_search(query)`

Залогиниться при необходимости → выполнить поиск ИТС → распарсить список
результатов → вернуть **структурный JSON**:

```json
[{ "title": "...", "snippet": "...", "ref": "db/v854doc#bookmark:...", "db": "v854doc" }]
```

`ref` пригоден для передачи в `its_read`. **Оговорка (S6):** поиск ИТС может быть
пообъектным (внутри базы), а не глобальным — тогда `its_search` требует/использует
`db`-скоуп. Уточняется спайком.

### 4.4. `its_read(ref)`

`ref` = полный URL или закладка (`db/v854doc#bookmark:adm:TI000001410`). Путь
получения контента определяет спайк:

- **(a) чистый XHR/print-URL** (SPA дёргает за содержимым секции; у ИТС часто есть
  «версия для печати» по стабильному URL) → берём HTML напрямую через `page.goto`
  в том же авторизованном контексте: быстро, минимальный след, менее «ботово».
  **Предпочтительно.**
- **(b) только SPA** → навигируем браузер по полному URL с хэшем, ждём контейнер
  контента, извлекаем.

Извлечение: срезать nav / toc / шапку / футер, взять основную панель → **markdown**
(заголовки, списки, таблицы, примеры кода) + канонический URL + заголовок.
Внутренние кросс-ссылки (сами `#bookmark:`-рефы) сохраняем — агент может ходить по
ним через повторный `its_read`.

**Картинки/диаграммы (S7).** Схемы и скриншоты в доках за авторизацией напрямую
агенту недоступны. v1: заменяем на плейсхолдеры `[изображение: <alt>]` в markdown
(не теряем факт наличия, теряем содержимое). Проксирование картинок через signed
URL (как uploads) — возможное улучшение следующей итерации, не v1. Зафиксировать
в выводе, что изображения опущены.

## 5. Анти-бот (детали)

- **Персистентная залогиненная сессия** — главный выигрыш.
- **Stealth** (см. 3.1).
- **Человеческий темп**: рандомизированные паузы между действиями, минимальный
  интервал между запросами, без параллелизма.
- **Уважение к сигналам**: `429/503` → экспоненциальный бэкофф; агрессивный кэш.
- **Консервативный перехват** (§4.1) — не воюем за сессию с человеком.
- **Жёсткая граница**: капча / 2FA / глухой блок → структурная ошибка оператору +
  нотификация; **не пытаемся ломать**.

## 6. Креды (S4 — решение)

- Хранятся в OPEX secrets vault: новый секрет `ITS_CREDENTIALS`
  (`{login, password}`), scope global. Не в `.env`, не в конфиге, не в git.
- Операторские креды кладём в vault при деплое (через secrets API или UI) —
  **в код/git не пишем**.
- **Доставка в toolgate — дефолт-решение:** toolgate уже обращается к core API на
  старте (загрузка реестра провайдеров, `toolgate/config.py`). Переиспользуем этот
  же канал toolgate↔core: toolgate запрашивает `ITS_CREDENTIALS` у внутреннего
  эндпоинта core (тем же путём аутентификации, что и остальной startup-обмен).
  Точный механизм/эндпоинт подтверждается **первой задачей плана** (проверить
  существующий auth-путь toolgate→core). Fallback, если канал не подходит:
  env-инжект секрета в процесс toolgate из vault при спавне (process_manager).
- `browser-renderer` креды не хранит — получает как значения действий `type` в шаге
  логина.

## 7. Кэш

- `its_read` по канонич. `ref` → markdown, TTL ~сутки (доки меняются редко).
- `its_search` по нормализованному запросу → результаты, TTL ~час.
- v1: in-memory LRU в toolgate + файловый кэш под томом профиля
  (`/profiles/its/cache`). Инвалидация — только по TTL.
- **Канонизация `ref` (S10):** нормализовать ключ кэша (алиасы, порядок,
  хэш-фрагмент), чтобы один и тот же контент не кэшировался под разными ключами и
  не устаревал вразнобой.

## 8. Сериализация и конкурентность

- Один персистентный контекст на профиль (лок `user-data-dir`) + `asyncio.Lock`
  в toolgate вокруг всей операции → все агенты ходят через одну сессию, FIFO, с
  троттлингом между операциями.
- **Hard per-op timeout + watchdog (S5).** Лок держится на всю браузерную операцию
  (секунды–минуты). Обязателен жёсткий таймаут операции: по истечении — операция
  прерывается, лок освобождается, агент получает `its_timeout`. Один зависший
  `read` не должен заклинивать инструмент для всех.
- **Латентность:** ITS-вызовы глобально сериализованы — холодный медленный `read`
  блокирует очередь. Для низкообъёмного инструмента приемлемо; кэш смягчает
  повторы.
- Контекст `its` живёт всё время жизни сервиса; страницы recycled.

## 9. Обработка ошибок

Структурно наверх, каждая — с внятным текстом, который агент передаст
пользователю:

- `its_busy` — сессия занята (консервативный перехват, §4.1);
- логин-фейл (неверные креды / капча / 2FA) → + нотификация оператору;
- `its_timeout` — превышен таймаут операции (§8);
- ИТС недоступен / таймаут сети;
- контент по `ref` не найден.

## 10. Тестирование

- **`browser-renderer` (generic):** pytest-юниты на логику персистентных профилей,
  «singleton-контекст на профиль», применение stealth — через fake-page (шаблон в
  `docker/browser-renderer/test_dispatch.py`).
- **toolgate ITS-модуль:** оффлайн-юниты против **HTML-фикстур**, снятых в спайке —
  извлечение → markdown, детект «разлогинено», парсинг результатов поиска, cooldown
  перелогина, таймаут. Клиент `browser-renderer` мокается. CI без живого ИТС.
- **Live E2E (руками, на сервере, реальные креды):** `login → its_search →
  its_read(v854doc TI000001410) → чистый markdown`. Приёмочный критерий.
- **Периодический live-smoke (S11):** оффлайн-фикстуры устаревают молча при смене
  DOM ИТС → предусмотреть лёгкий периодический live-прогон (cron/watchdog или ручной
  чеклист), чтобы ловить дрейф селекторов.

## 11. Деплой

- `browser-renderer`: пересборка Docker-образа (из `~/opex/docker/` на сервере) +
  монтирование тома `/profiles` в `docker-compose.yml`, чтобы куки пережили
  пересборку.
- toolgate: новый модуль + роуты; деплой синком `.py` + рестарт сервиса toolgate.
- YAML-tool: добавить `workspace/tools/its.yaml` (hot-load).
- Секрет `ITS_CREDENTIALS` в vault.

## 12. Phase 0 — Spike (обязательно первым)

На живых кредах разведать и задокументировать:

1. **Логин-флоу:** URL, селекторы формы, редиректы, есть ли **капча / 2FA /
   интерстишл «вошли в другом месте»** (определяет необходимость assisted-login,
   §4.2).
2. **Долговечность сессии (S8):** как долго живут куки, как часто нужен перелогин.
3. **Параллельные сессии (S1):** сколько одновременных сессий разрешает подписка
   (влияет на строгость консервативного перехвата).
4. **Контент `#bookmark:`:** есть ли чистый XHR / print-URL (путь a) или только SPA
   (путь b); снять HTML-фикстуру страницы.
5. **Поиск ИТС:** глобальный или пообъектный (S6), endpoint/URL, DOM результатов;
   снять фикстуру.
6. **Анти-бот:** что проходит по UA/темпу, что триггерит блок; хватает ли
   headless + stealth или нужен headful + xvfb.

**Выхлоп:** короткий findings-док + HTML-фикстуры, питающие unit-тесты извлечения.
Спайк снимает главный риск неизвестных механик; только после него фиксируем
селекторы, путь получения контента (a/b) и строгость перехвата.
