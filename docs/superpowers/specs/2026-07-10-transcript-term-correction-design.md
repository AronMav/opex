# Коррекция искажённых названий в авто-транскриптах (term_fixer)

**Дата:** 2026-07-10
**Статус:** дизайн утверждён, реализация не начата

## Проблема

STT искажает названия продуктов, брендов и терминов: «T-G37» вместо «Tape J-37»,
«амбассадор» вместо «MBassador», «тейп-обработка» вместо «Tape». Ошибки попадают
и в сырой транскрипт (`transcribe`), и в конспект (`summarize_video`), делая
названия непригодными для поиска и покупки упомянутых инструментов.

Метод исправления (проверен вручную):

1. **Контекст** — рядом с кривым именем в транскрипте описано, ЧТО объект делает
   («тейп-обработка от Arturia — дисторшн, плёнка плавает»).
2. **Фонетика + числа** — «тейп» ≈ Tape, «амбассадор» ≈ MBassador, общий код
   «37» ↔ «J-37».
3. **Веб-поиск по описанию**, а не по кривому имени: «Arturia tape plugin
   distortion flutter» → Arturia Tape J-37.
4. **Сверка с реальностью** — продукт существует и совпадает по функции.

## Решение

Новый этап «detect → search → verify → apply» между STT и выдачей/конспектом.
Выполняется **один раз над транскриптом** — исправленные названия автоматически
попадают и в конспект, и во вложенный в заметку транскрипт.

Уверенные исправления подставляются молча; неуверенные помечаются в тексте
(«T-G37 *(вероятно Tape J-37?)*»). Все замены перечисляются в глоссарии.

**Вне скоупа:** проверка, установлены ли найденные плагины у пользователя
(обработчик toolgate не имеет доступа к ПК пользователя); vision-обогащение;
исправление обычных слов (не названий).

## Архитектура

### Модуль `toolgate/term_fixer.py`

Новый модуль на корне toolgate (рядом с `video_helpers.py`); обработчики
импортируют его через тот же sys.path-шов, что и `video_helpers`.

```python
@dataclass
class FixResult:
    transcript: str          # исправленный текст (или исходный, если замен нет)
    replacements: list[dict] # [{heard, variants, corrected, confidence, description}]
    glossary_md: str         # готовый markdown-блок; "" если замен нет

async def fix_terms(ctx, transcript: str, language: str = "ru") -> FixResult
```

`MAX_CANDIDATES = 8` — кап кандидатов на один транскрипт.

### Шаг 1 — Detect (1 вызов `ctx.llm.complete`)

Промпт: «Ниже авто-транскрипт. Найди названия продуктов/брендов/моделей/терминов,
которые выглядят как фонетическое искажение (несуществующие названия, кириллица
там, где ожидается латинский бренд, странные буквенно-цифровые коды). Для
каждого верни JSON: точная форма из текста (`heard`), все словоформы,
встречающиеся в тексте (`variants`), функциональное описание из контекста
(`description` — что объект делает), поисковый запрос на английском ПО
ОПИСАНИЮ, а не по кривому имени (`query`)».

Выход — JSON-массив. Пустой массив → немедленный выход (`FixResult` с исходным
текстом); оверхед на чистом транскрипте = один LLM-вызов.

### Шаг 2 — Search

Для каждого кандидата (первые `MAX_CANDIDATES`) один
`ctx.search(query, max_results=5)`. К кандидату прикладываются title + snippet +
url результатов.

### Шаг 3 — Verify (1 батч-вызов LLM на все кандидаты)

Промпт: «Для каждого кандидата: услышано X, описание Y, выдача поиска Z. Реши,
какое реальное название имелось в виду. Критерии: фонетическое сходство
(тейп≈Tape, амбассадор≈MBassador), общие цифры/коды (37↔J-37), совпадение
бренда и функции с описанием из контекста. `confidence: high` только когда
сходятся и фонетика, и функция; иначе `low`. Если подходящего продукта в выдаче
нет — `corrected: null`» (кандидат отбрасывается).

### Шаг 4 — Apply (чистый Python, без LLM)

- `high` — замена **всех** вхождений всех `variants` на `corrected`.
- `low` — аннотация только **первого** вхождения:
  `амбассадор (вероятно MBassador?)`; остальные вхождения не трогаются
  (не заспамить длинный транскрипт).
- Порядок замен: от длинных словоформ к коротким («амбассадора» раньше
  «амбассадор» — иначе получится «MBassadorа»).
- No-op guard: `corrected == heard` → замена не применяется.
- Словоформа не найдена в тексте → ноль замен, не ошибка; кандидат, ни одна
  словоформа которого не найдена, не попадает в глоссарий.
- Таймкоды `[MM:SS]` не затрагиваются (замена по словоформам).

Транскрипт любой длины через LLM целиком НЕ прогоняется: detect получает
транскрипт (он и так влезает в конспект-промпты), verify — только кандидатов и
сниппеты, замена детерминированная. Цена этапа: 2 LLM-вызова + ≤8 поисков.

### Глоссарий

```markdown
## Исправленные названия
- «амбассадор» → **MBassador** (MeldaProduction) — суб-бас плагин
- «T-G37» → *вероятно* **Arturia Tape J-37** (не подтверждено)
```

## Интеграция

### Valve (оба обработчика)

```xml
<field name="fix_terms" type="bool" default="true" label="Исправлять названия"
       description="Определять искажённые STT названия (бренды, плагины, термины) и исправлять их через веб-поиск"/>
```

Чтение — как `include_transcript`:
`str(ctx.config.get("fix_terms") or "true").lower() not in ("false", "0", "no")`.
Никаких `params`-переопределений (YAGNI).

### transcribe.py

После empty-guard, перед выдачей (valve on):

```python
await ctx.progress("fix_terms", 70)
fx = await fix_terms(ctx, text, language)
return ctx.result.text(fx.transcript + ("\n\n" + fx.glossary_md if fx.glossary_md else ""))
```

### summarize_video.py

Сразу после empty-guard, до digest:

```python
await ctx.progress("fix_terms", 40)
fx = await fix_terms(ctx, transcript, language)
transcript = fx.transcript
```

Конспект строится по исправленному транскрипту (правильные названия и
low-confidence аннотации «(вероятно X?)» попадают в конспект естественно);
вложенный транскрипт в заметке — исправленный. `glossary_md` передаётся в
`build_note` новым опциональным параметром `glossary: str = ""` и вставляется
**после тела конспекта, перед свёрнутым транскриптом**.

### Прогресс / UI

Новая фаза `fix_terms`:

- `PHASES` в [ui/src/components/chat/VideoProgressIndicator.tsx](../../../ui/src/components/chat/VideoProgressIndicator.tsx)
  — запись `fix_terms: { emoji: "🔎", key: "chat.video_phase_fix_terms" }`.
- `ru.json`: «Проверяю названия…»; `en.json`: «Verifying names…».

Core/channels не меняются: фаза проходит сырым ключом через WS-событие
`file_job_progress`; неизвестные фазы старый UI показывает сырым текстом
(fail-soft уже встроен).

## Обработка ошибок

Базовый принцип: **коррекция никогда не роняет джобу** — любая ошибка внутри
`fix_terms` логируется warning-ом и возвращает исходный транскрипт без замен.

| Сбой | Поведение |
| --- | --- |
| Нет активного websearch-провайдера | Этап пропускается целиком (`ctx.log.warning`). Без сверки с реальностью подстановки не делаем — LLM-догадки без поиска опаснее исходных ошибок. |
| Detect вернул невалидный JSON | Парсер снимает markdown-заборы (```json), пробует `json.loads`; при неудаче — warning и исходный текст. Без повторных запросов. |
| Ошибка одного `ctx.search` | Кандидат уходит в verify с пометкой «без результатов»; verify отбрасывает неподтверждённых (`corrected: null`). |
| Verify упал / вернул мусор | Warning, исходный текст. |
| LLM выдумал словоформу | Ноль замен по ней; не ошибка. |

## Тестирование (TDD — тесты первыми)

`toolgate/tests/`, по образцу существующих тестов обработчиков со стаб-ctx.

1. **Чистые функции:** парсинг detect-JSON (валидный / с заборами / мусор);
   `apply_replacements` — все вхождения при high, только первое при low,
   порядок длинные→короткие, сохранность таймкодов, no-op guard; сборка
   глоссария.
2. **Оркестратор `fix_terms`** со стабами `ctx.llm` / `ctx.search`: happy-path
   (замена + глоссарий); пустой detect; нет websearch-провайдера (скип);
   невалидный JSON (fail-soft).
3. **Интеграция обработчиков** со стабом `fix_terms`: valve on/off; глоссарий в
   выводе `transcribe`; глоссарий в заметке `summarize_video` (до
   транскрипт-блока).

Rust-тесты не затрагиваются. Python-тесты toolgate гоняются локально и на
сервере (venv).

существующему UI-деплой-пути (`deploy-ui.sh`).
