# Agent Audit — Аудит Агентов

## Назначение

Систематический анализ агентов системы OPEX: состояние, конфигурации, сессии, поведенческие паттерны, ошибки и аномалии. Навык специализирует классификационный фреймворк `discovery-protocol` (Level 0–3) на домен аудита агентов.

## Классификация задачи аудита (из discovery-protocol)

Перед началом любого аудита агента — классифицируй задачу по дереву `discovery-protocol`:

```
Аудит входит в известный паттерн?
├── YES: Этот тип аудита уже выполнялся для этого агента?
│   ├── YES ──> Level 0 — выполнить по известной схеме, сверить с предыдущим отчётом
│   └── NO: Известна схема аудита для другого агента?
│       ├── YES ──> Level 1 — адаптировать схему, проверить 2–3 файла конфигурации
│       └── NO ──> Level 2 — спроектировать процедуру аудита с нуля
└── NO: Понятен ли домен (архитектура агентов, Core API)?
    ├── YES ──> Level 2 — исследовать подход, затем выполнить
    └── NO ──> Level 3 — запросить разъяснения, не выполнять вслепую
```

### Level 0 — Рутинный аудит

**Сигнал:** Аудит этого агента уже выполнялся. Известны метрики, есть предыдущий отчёт.

**Действие:** Повторить схему, сравнить с предыдущими результатами, выдать дельту.

**Примеры:** Ежедневная проверка состояния агентов, мониторинг после планового перезапуска.

### Level 1 — Адаптация известной схемы

**Сигнал:** Аудит агента не выполнялся, но схема для аналогичного агента существует.

**Действие:** Загрузить конфигурацию агента (Core API), проверить 2–3 сессии, применить шаблон.

**Примеры:** Аудит нового агента, созданного по образцу существующего.

### Level 2 — Исследовательский аудит

**Сигнал:** Неизвестный агент, нестандартная конфигурация, аномалия неясного происхождения.

**Действие:** Полный цикл: сбор данных → исследование → анализ → отчёт.

**Примеры:** Агент с нестандартной цепочкой инструментов, миграция провайдера.

### Level 3 — Домен вне компетенции

**Сигнал:** Внешний сервис, незнакомый протокол, регуляторные требования.

**Действие:** Остановиться. Запросить разъяснения у пользователя.

## Процедура аудита

### Шаг 1 — Определение типа аудита

| Тип | Признак | Ключевой инструмент |
|-----|---------|---------------------|
| **Конфигурационный** | «проверь настройки агента», «какая модель у X» | `agents_list` |
| **Сессионный** | «что делал агент», «покажи сессии агента X» | `session(action="list")` + `session(action="history")` |
| **Поведенческий** | «почему агент так ответил», «найди ошибки агента» | `session(action="search")` + `memory(action="search")` |
| **Системный** | «агент не запускается», «статус агента», «канал агента» | `agents_list` + `agent(action="status")` |
| **Полный аудит** | «полный аудит агента X» | Все инструменты последовательно |

### Шаг 2 — Сбор данных

#### Конфигурационный аудит

```
1. agents_list → все агенты с provider, model, status
2. Для выбранного агента: запросить Core API GET /api/agents/{name}
3. Проверить: provider активен? model существует? channel настроен?
```

#### Сессионный аудит

```
1. session(action="list", limit=N) → список сессий
2. Фильтр по агенту: session(action="list", channel="agent") → grep по имени
3. Для целевых сессий: session(action="history", session_id="...")
4. Параллельно: session(action="search", query="имя агента")
```

#### Поведенческий аудит

```
1. session(action="search", query="[ключевое слово или ошибка]")
2. Для найденных сессий: session(action="history", ...)
3. memory(action="search", query="[агент] [тема]")
4. Выявить паттерны: частые ошибки, токсичные паттерны, повторяющиеся сбои
```

#### Системный аудит

```
1. agents_list → текущий статус
2. agent(action="status", target="имя_агента") → состояние экземпляра
3. Проверить Core API: GET /api/agents/{name}/channels
4. Проверить: provider назначен? (GET /api/profiles — слоты капабилити; GET /api/provider-active показывает только embedding)
```

### Шаг 3 — Анализ

Критерии оценки агента:

| Аспект | Что проверять | Признак проблемы |
|--------|---------------|------------------|
| **Конфигурация** | provider, model, channel, SOUL.md | provider отключён, model не существует, channel не активен |
| **Активность** | сессии за период, частота вызовов | ноль сессий (агент не используется), аномальный всплеск |
| **Качество ответов** | ошибки инструментов, таймауты, пустые ответы | частые отказы инструментов, токсичные паттерны |
| **Поведение** | соответствие SOUL.md и IDENTITY.md | ответы не соответствуют стилю, нарушение ролей |
| **Ресурсы** | количество сессий, частота heartbeats | утечка сессий, чрезмерное потребление токенов |

### Шаг 4 — Отчёт

Формат отчёта зависит от уровня классификации:

#### Level 0–1 (краткий)

```
## Аудит агента: [имя]

- Статус: [OK / WARN / CRIT]
- Provider: [имя], model: [модель], статус: [активен/проблема]
- Channel: [тип], статус: [активен/проблема]
- Сессии (24ч): [N]
- Изменения с прошлого аудита: [дельта]
```

#### Level 2–3 (полный)

```
## Полный аудит агента: [имя]
**Дата:** [timestamp]
**Классификация:** Level [N]
**Исполнитель:** [agent]

### 1. Конфигурация
- Provider: ...
- Model: ...
- Channel: ...
- SOUL.md/IDENTITY.md: [соответствуют / отклонения]

### 2. Активность
- Всего сессий: N
- За 24ч: N
- Тренд: [стабильно / рост / падение]

### 3. Анализ сессий
- Проверено сессий: N
- Ошибок инструментов: N
- Паттерны: [...]

### 4. Поведенческий анализ
- Соответствие SOUL.md: [%]
- Отклонения: [...]

### 5. Вердикт
- Общее состояние: [OK / WARN / CRIT]
- Рекомендации: [...]

### 6. Сравнение с предыдущим аудитом
- [дельта]
```

### Шаг 5 — Фиксация результатов

После завершения аудита:

```
memory(action="index", content="[ключевые находки]", source="agent-audit", shared=true)
```

Для критических находок — немедленно уведомить пользователя через `message`.

## Правила

1. **Классифицируй перед началом.** Первый вопрос: «Какой это Level по discovery-protocol?» Ответ определяет глубину.
2. **Не перечитывай свои файлы.** SOUL.md, IDENTITY.md, AGENTS.md загружаются автоматически.
3. **Не дублируй.** Если аудит того же агента выполнялся недавно — сошлись на предыдущий, добавь только дельту.
4. **Проверяй provider-active.** Агент с отключённым провайдером — это критическая аномалия, а не рядовая ситуация.
5. **Экономия токенов.** Для рутинного Level 0 — только сводка. Не читай 50 сессий ради статуса «OK».
6. **Рефереальные петли.** Если сам агент-аудитор выполняет аудит самого себя — ограничиться конфигурационной проверкой, не анализировать собственные сессии рекурсивно.
7. **Приоритет критичности.** CRIT (агент неработоспособен) > WARN (отклонения) > OK. При CRIT — немедленный отчёт пользователю.

## Интеграция с discovery-protocol

Данный навык наследует:

- **Дерево решений** (Level 0–3) — адаптировано под домен аудита агентов
- **Правила реклассификации** — upgrade/downgrade при изменении оценки сложности
- **Бюджеты времени** — 0 мин (L0), 2–5 мин (L1), 15–30 мин (L2), 1ч+ (L3)
- **Anti-patterns** — примеры ошибок классификации при аудите агентов

Отличия от родительского навыка:

- Добавлены домен-специфичные триггеры (русские и английские)
- Процедуры и форматы отчётов адаптированы под агентов системы
- Введено понятие рефереальной петли (аудит агентом самого себя)
- Инструменты ограничены релевантными: `agents_list`, `session`, `agent`, `memory`
